## 1. 인덱싱 전략

2025년, 2026년 두 PDF를 각각 `PDFPlumberLoader`로 로드하고, 문서 로드 직후 각 청크에 `source_year` 메타데이터를 부여했다. 이후 `RecursiveCharacterTextSplitter`로 청킹한 뒤, 두 연도의 청크를 합산하여 하나의 ChromaDB에 인덱싱했다.

```python
CHUNKING_SIZE = 500
CHUNKING_OVERLAP = 100
```

메타데이터 구조 예시:
```python
doc.metadata["source_year"] = "2025"  # 또는 "2026"
```

ChromaDB가 이미 존재하면 재구축 없이 로드하고, BM25는 DB에 persist되지 않으므로 매 실행마다 원본 청크로 재구축했다.

## 2. Golden Dataset 설계

총 20문항을 직접 구축했다. 구성 방향은 다음과 같다.

- **연도별 대칭 구성**: 동일 주제에 대해 2025년 / 2026년 각각 물어보는 문항 쌍을 만들어, 연도 혼동 여부를 직접 관측할 수 있도록 했다 (예: q01/q02, q03/q04, q06/q07 등)
- **변경 사항 중심 문항**: 2025→2026 사이에 실제로 수치가 바뀐 항목(정신질환 장기지속형 주사제 본인부담률, 조산아 지원기간, 노숙인 시설 유효기간 등)을 의도적으로 포함시켜 연도 혼동이 오답으로 직결되도록 설계했다
- **교차 비교(cross-year) 문항**: q13, q14, q20 세 문항은 두 연도의 정보를 동시에 검색해야 정답이 가능한 고난이도 문항이다
- **난이도 분포**: easy 8문항, medium 4문항, hard 5문항, cross-year 3문항

## 3. Basic RAG 파이프라인 구성 (step1_basic_rag.py)

**Retriever 설정**
- `vectorstore.as_retriever(search_kwargs={"k": 5})` — 메타데이터 필터 없이 단순 벡터 유사도 Top-5 검색

**프롬프트 템플릿**
```
아래 컨텍스트를 바탕으로 질문에 답하세요.
각 컨텍스트에는 출처 년도(source_year)가 포함되어 있습니다. 질문이 묻는 년도와 일치하는 정보를 우선적으로 사용하세요.
정보가 부족하면 "정보를 찾을 수 없습니다"라고 답하세요.

컨텍스트:
{context}

질문: {question}

답변:
```

컨텍스트 포맷팅 시 `[출처: {source_year}년]` 접두사를 붙여 LLM이 연도를 인식할 수 있도록 했다.

**LLM 설정**: `ChatOpenAI(model="gpt-4o", temperature=0)`

**평가 방식**: `expected_answer`와 LLM 답변 모두 공백·개행을 제거한 후 포함 여부로 자동 판정

## 4. Advanced RAG 파이프라인 구성 (step2, step3)

### step2_hybrid_filter.py — Hybrid Search + 메타데이터 필터링

**Hybrid Search 설정**

| 항목 | 설정값 |
|------|--------|
| BM25 Retriever k | 5 |
| Vector Retriever k | 5 |
| Ensemble 가중치 (BM25 : Vector) | 0.5 : 0.5 |
| 메타데이터 필터링 | 질문에 연도 명시 시 자동 적용 |

질문 내 "2025"/"2026" 문자열 존재 여부로 연도를 동적으로 추출하여, 해당 연도의 청크만 벡터 검색 대상으로 제한했다. 교차 비교 문항처럼 두 연도가 모두 언급되거나 미언급인 경우에는 필터를 걸지 않았다.

### step3_reranking.py — Hybrid Search + 메타데이터 필터링 + Cohere Re-ranking

step2 설정에 추가로 아래를 적용했다.

| 항목 | 설정값 |
|------|--------|
| BM25 k / Vector k | 10 / 10 (1차 후보 확대) |
| Re-ranker 모델 | Cohere rerank-v3.5 |
| Re-ranking 후 최종 Top-K | 3 |

Hybrid Search로 넓게(10+10개) 후보를 수집한 뒤, Cohere Rerank로 질문과의 관련도를 재스코어링하여 상위 3개만 LLM에 전달했다. Cohere 무료 티어 한도(1분 10회) 방지를 위해 문항 간 6.5초 대기를 삽입했다.

### 인사이트

**BM25의 결정적 기여**: Basic RAG 오답 7건 중 6건이 Hybrid Search 도입으로 개선됐다. 공통 패턴은 의료 도메인 고유 용어("조산아", "이상지질혈증", "잠복결핵", "KTAS", "유효기간" 등)였다. 벡터 검색은 의미 공간에서 근접하나 해당 키워드가 없는 청크를 반환했고, BM25는 정확한 표면 매칭으로 이를 보완했다.

**메타데이터 필터링의 연도 혼동 해결**: 2025/2026 문서는 구조와 표현이 거의 동일하기 때문에 벡터 유사도만으로는 연도 구분이 불가능하다. 필터링으로 검색 공간을 제한하는 것이 가장 직접적이고 효과적인 해결책이었다.

**Re-ranking의 한계와 역할**: 정답률 측면에서 step2와 step3의 차이는 없었다. 그러나 q07처럼 Re-ranker가 의미적으로 관련된 것처럼 보이지만 실제 정답을 담지 않은 청크(365회 초과 본인부담률)를 1위로 올리는 사례가 있었다. 이번 데이터셋처럼 후보가 이미 필터링으로 정제된 경우, Re-ranking의 추가 이득이 제한적일 수 있다. Re-ranking은 후보가 많고 노이즈가 클 때 효과가 두드러진다.

## 8. 이론 과제 답변

### 8-1. Hybrid Search란?

Hybrid Search는 벡터 검색(Semantic Search)과 키워드 검색(BM25)을 결합하는 방식이다.

**왜 벡터 검색만으로 부족한가**: 벡터 검색은 의미적 유사성 기반이므로, 정확한 키워드를 포함하지 않아도 의미가 비슷한 청크를 가져온다. 반대로 "이상지질혈증", "KTAS 4", "잠복결핵"처럼 도메인 특수 용어는 학습 분포에서 희소하여 임베딩 벡터 공간에서 의미적으로 잘 분리되지 않을 수 있다. 결과적으로 정확한 용어가 포함된 문서를 놓칠 수 있다.

| 검색 방식 | 강점 | 약점 |
|----------|------|------|
| 벡터 검색 (Semantic) | 표현이 달라도 의미가 같으면 검색, 동의어·문맥 이해 | 정확한 키워드 매칭 실패, 도메인 특수 용어 처리 취약 |
| BM25 (Keyword) | 정확한 토큰 매칭, 도메인 용어 강점, 속도 빠름 | 동의어·문맥 이해 불가, 표현이 조금 달라지면 누락 |
| Hybrid (결합) | 두 방식의 약점을 상호 보완, 재현율과 정밀도 균형 | 결합 가중치 튜닝 필요, 구현 복잡도 증가 |

**결합 방식**: 이번 과제에서는 `EnsembleRetriever`(가중치 기반 RRF 변형)를 사용했다. 각 리트리버의 결과를 순위 기반으로 병합하는 **Reciprocal Rank Fusion(RRF)** 방식이 가중치 기반보다 파라미터 민감도가 낮아 실무에서 자주 사용된다.

**BM25 알고리즘 핵심**: TF-IDF를 개선한 확률 모델로, 문서 길이 정규화(`b` 파라미터)와 TF 포화 함수(`k1` 파라미터)를 적용하여 긴 문서나 반복 단어에 대한 과적합을 방지한다.

### 8-2. Re-ranking이란?

Re-ranking은 1차 검색(Retrieval)으로 가져온 후보 문서들을 질문과의 정밀한 관련도로 재정렬하는 Post-Retrieval 기법이다. RAG 파이프라인에서 Retrieval과 Generation 사이에 위치한다.

**Bi-encoder vs Cross-encoder**

| 구분 | Bi-encoder | Cross-encoder |
|------|-----------|---------------|
| 입력 방식 | 질문과 문서를 **각각** 인코딩 | 질문과 문서를 **함께** 인코딩 |
| 속도 | 빠름 (사전 임베딩 가능) | 느림 (쌍마다 새로 추론) |
| 정확도 | 상대적으로 낮음 | 높음 (상호 주의로 세밀한 관계 포착) |
| 사용 위치 | 1차 검색 (대규모 후보 선별) | Re-ranking (소수 후보 정밀 재정렬) |

Cross-encoder를 검색 단계에서 쓰지 않는 이유는 N개의 후보 전체에 대해 쿼리와의 쌍을 실시간으로 추론해야 하므로 전체 문서 규모에 적용하면 지연이 극심해지기 때문이다. Two-stage 패턴(Bi-encoder로 후보 수십 개 선별 → Cross-encoder로 최종 Top-K 선택)이 속도와 정밀도의 균형을 달성한다.

**Cohere Rerank**: 상용 API로 100개 이상 언어를 지원하며, `rerank-v3.5` 모델은 다국어 문서-쿼리 관련도를 0~1 사이 relevance_score로 반환한다. 무료 티어 월 1,000회로 소규모 평가에 충분하다.

### 8-3. 메타데이터 필터링과 컨텍스트 압축

**메타데이터 필터링**

한 줄 정의: 청크에 부여된 메타데이터(연도, 출처, 카테고리 등)를 기반으로 검색 범위를 사전에 제한하는 기법.

없으면 어떤 문제가 발생하는가: 이번 과제처럼 구조가 동일한 다년도 문서를 동시에 인덱싱하면, 벡터 유사도만으로는 연도를 구분할 수 없다. "2026년 본인부담률"을 물었을 때 2025년 청크가 상위에 오를 수 있고, LLM은 컨텍스트만 보므로 틀린 연도의 답을 생성한다.

LangChain Chroma 구현 예시:
```python
vector_retriever = vectorstore.as_retriever(
    search_kwargs={"k": 5, "filter": {"source_year": "2026"}}
)
```

**컨텍스트 압축 (Contextual Compression)**

한 줄 정의: 검색된 청크에서 질문과 관련 있는 부분만 추출·압축하여 LLM에 전달하는 Post-Retrieval 기법.

없으면 어떤 문제가 발생하는가: 청크 전체를 LLM에 전달하면 질문과 무관한 내용이 컨텍스트를 오염시켜 정답률이 낮아지고 토큰 비용이 증가한다. 특히 500자 청크에 핵심 수치가 한 줄뿐인 경우가 많다.

LangChain 구현 예시:
```python
from langchain.retrievers import ContextualCompressionRetriever
from langchain_cohere import CohereRerank

compressor = CohereRerank(model="rerank-v3.5", top_n=3)
compression_retriever = ContextualCompressionRetriever(
    base_compressor=compressor,
    base_retriever=ensemble_retriever
)
```

### 8-4. 실습 전 가설 vs 실제 결과

**가설 1**: 2025/2026 연도 혼동은 수치가 바뀐 항목(정신질환 주사제 본인부담률 5%→2% 등)에서 집중 발생할 것이다.
**실제**: 연도 혼동보다 검색 누락(키워드 매칭 실패)이 오답의 주된 원인이었다. BM25 도입 후 해당 문항들이 대부분 개선됐다.

**가설 2**: Hybrid Search는 "이상지질혈증", "KTAS 4" 같은 전문 용어 포함 문항에서 가장 효과적일 것이다.
**실제**: 예측 적중. q06, q07, q09, q19가 모두 도메인 특수 용어 매칭 실패였고 BM25로 해결됐다.

**가설 3**: Re-ranking이 결과를 악화시킬 수 있는 경우가 있을 것이다.
**실제**: q07에서 Re-ranker가 365회 초과 청크(관련 주제이나 다른 내용)를 1위로 배치하는 준-악화 사례가 관찰됐다. 정답률에는 영향 없었지만, 메타데이터 필터링으로 이미 후보가 정제된 상황에서는 Re-ranking의 부가 가치가 제한적이었다.

**가설 4**: 메타데이터 필터링이 연도 혼동 문제를 대부분 해결할 것이다.
**실제**: 예측 적중. 연도가 명시된 단일 연도 문항에서 필터링은 연도 혼동을 완전히 차단했다. cross-year 문항은 필터를 걸 수 없어 BM25 보완에 의존했다.
