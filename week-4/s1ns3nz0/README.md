## 1. 이론 과제

### 1-1. Hybrid Search란?

**정의:** 벡터 검색(Semantic Search)과 키워드 검색(BM25)을 결합하여, 각각의 약점을 보완하는 검색 방식.

**왜 벡터 검색만으로 부족한가?**
- 벡터 검색은 텍스트의 "의미"를 벡터로 변환하여 유사도를 측정한다
- 그러나 정확한 키워드(예: "추나요법", "KTAS", "1,000원")를 포함한 문서를 놓칠 수 있다
- 특히 PDF 표 데이터가 깨진 경우, 의미적 유사도가 낮아져 벡터 검색이 실패한다

**BM25 알고리즘의 핵심 원리:**
- TF-IDF 기반으로, 질문에 포함된 단어가 문서에 몇 번 등장하는지(TF)와 전체 문서에서 얼마나 희귀한지(IDF)를 계산
- 정확한 토큰 매칭으로 작동하므로, 깨진 표 텍스트에서도 "1종", "외래", "1,000원" 같은 키워드를 찾을 수 있다

| 검색 방식 | 강점 | 약점 |
|----------|------|------|
| 벡터 검색 (Semantic) | 의미적으로 유사한 문서 발견, 동의어/패러프레이즈 처리 | 정확한 키워드 매칭 실패, 도메인 전문 용어 취약 |
| BM25 (Keyword) | 정확한 용어/숫자 매칭, 도메인 용어에 강함 | 동의어 처리 불가, 질문 표현이 다르면 실패 |
| Hybrid (결합) | 양쪽 장점 결합, 검색 범위 확대 | 구현 복잡도 증가, 가중치 튜닝 필요 |

**결합 방식:** 본 과제에서는 Vector Top-10 + BM25 Top-10을 가져온 후 중복 제거하여 병합했다. 가중치는 0.5:0.5 (동일 비중)으로 설정했다.

### 1-2. Re-ranking이란?

**정의:** 1차 검색(Retrieval)으로 가져온 후보 문서들을 더 정밀한 모델로 재정렬하여, 질문과 가장 관련 높은 문서를 상위로 올리는 기법. RAG 파이프라인에서 Retrieval과 Generation 사이에 위치한다.

| 구분 | Bi-encoder | Cross-encoder |
|------|-----------|---------------|
| 입력 방식 | 질문과 문서를 **각각** 인코딩 | 질문과 문서를 **함께** 인코딩 |
| 속도 | 빠름 (벡터 미리 계산 가능) | 느림 (매번 쌍으로 계산) |
| 정확도 | 상대적으로 낮음 | 높음 (문맥 상호작용 반영) |
| 사용 위치 | 1차 검색 (대규모 후보 선별) | Re-ranking (소수 후보 재정렬) |

**왜 Cross-encoder를 검색 단계가 아닌 Re-ranking에서 사용하는가?**
- Cross-encoder는 (질문, 문서) 쌍을 함께 인코딩하므로 정확도가 높지만, 146개 청크 모두에 대해 실행하면 너무 느리다
- 따라서 Bi-encoder(벡터 검색)로 빠르게 후보 20개를 선별한 후, Cross-encoder로 20개만 재정렬하는 **Two-stage retrieval** 패턴을 사용한다

**Cohere Rerank의 동작 방식:**
1. 질문과 각 후보 문서를 쌍으로 입력
2. Cross-encoder가 각 쌍의 관련성 점수를 계산
3. 점수 기준으로 재정렬하여 상위 N개 반환

### 1-3. 메타데이터 필터링과 컨텍스트 압축

| 기법 | 정의 | 왜 필요한가 | 구현 방식 |
|------|------|-----------|----------|
| **메타데이터 필터링** | 청크에 포함된 메타데이터(출처 년도, 카테고리 등)를 기반으로 검색 범위를 제한하는 기법 | 다년도 문서 RAG에서 년도 혼동 방지. 필터링 없이는 2025년 질문에 2026년 청크가 섞여 오답 발생 | Chroma: `search_kwargs={"filter": {"source_year": "2025"}}` |
| **Contextual Compression** | 검색된 청크에서 질문과 관련 있는 부분만 추출하여 LLM에 전달하는 기법 | 긴 청크에 무관한 정보가 많으면 LLM이 혼동. 관련 부분만 전달하면 정확도 향상 | LangChain `ContextualCompressionRetriever` + `LLMChainExtractor` |

본 과제에서는 메타데이터 필터링 대신 **프롬프트에 년도 출처를 명시**하는 방식으로 년도 인식을 구현했다. 각 컨텍스트 앞에 `[출처: 2025년 문서]`를 붙여 LLM이 년도를 구분하도록 했다.

## 2. 인덱싱 전략

| 항목 | 설정 |
|------|------|
| PDF 로더 | PyPDFLoader |
| Text Splitter | RecursiveCharacterTextSplitter |
| chunk_size | 500 |
| chunk_overlap | 100 |
| 2025년 청크 수 | 73개 |
| 2026년 청크 수 | 73개 |
| 총 청크 수 | 146개 |
| 메타데이터 | `source_year` (각 청크에 "2025" 또는 "2026" 태깅) |

**두 년도 문서의 인덱싱 방식:** 하나의 FAISS 벡터 저장소에 함께 저장하되, 각 청크의 metadata에 `source_year` 필드를 추가하여 출처 년도를 구분할 수 있도록 했다. RAG 프롬프트에서 각 컨텍스트 앞에 년도를 명시하여 LLM이 년도를 인식하도록 했다.

## 3. Golden Dataset 설계

### 구성

| 난이도 | 문항 수 | 설계 의도 |
|--------|--------|---------|
| easy | 10 | 표에서 단일 조건으로 찾을 수 있는 값 (1종 외래 1,000원, 틀니 5% 등) |
| medium | 10 | 2~3개 조건 조합 또는 본문 텍스트에서 찾아야 하는 정보 |
| hard | 1 | 복잡한 조건 (응급의료관리료 KTAS 기준) |
| cross-year | 4 | 2025년과 2026년을 비교해야 하는 문항 |
| **합계** | **24** | |

### 교차 비교 문항 설계 의도

두 PDF는 구조와 표현이 ~90% 동일하여 RAG 시스템이 년도를 혼동하기 매우 쉽다. 실제로 확인한 **년도 간 차이점**을 기반으로 cross-year 문항을 설계했다:

| 항목 | 2025년 | 2026년 |
|------|--------|--------|
| 장기지속형 주사제 본인부담률 | 5% | **2%** |
| 노숙인진료시설 유효기간 | 1년 연장 (~2025.3.21) | **3년** 연장 (~2028.3.21) |
| 외래 365회 초과 본인부담률 | 없음 | **30%** (신설) |
| 본인부담금 환급 산정기간 | 30일 | **월단위** |
| 조산아 지원 기간 | 5년 | **5년 4개월** |
| 응급의료관리료 기준 | 응급증상 아닌 경우 | **KTAS 4·5** |

## 4. Basic RAG 파이프라인 구성

### 파이프라인 흐름

```
질문 → 임베딩 → FAISS 벡터 검색 (Top-5) → 컨텍스트 구성 (년도 라벨 포함) → Claude → 답변
```

### 프롬프트 템플릿

```
아래 컨텍스트를 바탕으로 질문에 답하세요.
각 컨텍스트에는 출처 년도가 표시되어 있습니다. 질문이 특정 년도를 묻는 경우 해당 년도의 정보만 사용하세요.
컨텍스트에 없는 내용은 "정보를 찾을 수 없습니다"라고 답하세요.
답변은 간결하게 핵심만 답하세요.

컨텍스트:
[출처: 2025년 문서, p.9]
{chunk_content}
---
[출처: 2026년 문서, p.9]
{chunk_content}

질문: {question}
답변:
```

### Retriever 설정

| 항목 | 설정 |
|------|------|
| 검색 방식 | FAISS similarity_search |
| Top-K | 5 |
| 임베딩 모델 | intfloat/multilingual-e5-small |

## 5. Advanced RAG 파이프라인 구성

### 파이프라인 흐름

```
질문 → [Vector Search Top-10] + [BM25 Search Top-10] → 중복 제거 → Cohere Re-rank → Top-5 → Claude → 답변
```

### 설정

| 항목 | 설정값 |
|------|--------|
| BM25 Retriever k | 10 |
| Vector Retriever k | 10 |
| Ensemble 가중치 (vector : BM25) | 0.5 : 0.5 |
| Re-ranker 종류 및 모델명 | Cohere rerank-v3.5 |
| Re-ranking 후 최종 Top-K | 5 |

### 메타데이터 필터링

명시적 메타데이터 필터링은 적용하지 않았다. 대신 프롬프트에 `[출처: {year}년 문서]`를 포함하여 LLM이 년도를 구분하도록 했다. 이 방식으로 Basic RAG에서 년도 검색 정확도 100%, Advanced RAG에서 95.8%를 달성했다.

### 8-3. 기법별 기여도 분석

| 방식 | 자동 정답률 | 수동 정답률 (추정) |
|------|-----------|-----------------|
| Basic RAG (벡터만) | 12/24 (50.0%) | ~18/24 (~75%) |
| + Hybrid Search (벡터 + BM25) | 15/24 (62.5%) | ~20/24 (~83%) |
| + Hybrid Search + Re-ranking | 16/24 (66.7%) | ~20/24 (~83%) |

**BM25의 기여:** +3~4 문항 개선. 특히 **표 데이터 검색**에서 결정적 역할. PyPDFLoader가 표 구조를 파괴해도 키워드("1종", "1,000원" 등)는 텍스트에 남아있어 BM25가 찾을 수 있었다.

**Re-ranker의 기여:** +1 문항 개선 (q08). Hybrid Search가 가져온 다수의 후보 중 가장 관련성 높은 청크를 상위로 올려 LLM의 답변 품질을 높였다. 수동 판정 기준으로는 답변의 "정확도"가 전반적으로 향상되었다 (예: "0%(면제)" → "무료(본인부담 면제)").

### 8-4. 핵심 인사이트

1. **BM25는 깨진 표 데이터에 가장 효과적이다.** 벡터 검색이 의미적으로 유사한 청크를 찾는 반면, BM25는 정확한 키워드를 포함한 청크를 찾는다. 의료급여제도처럼 정확한 숫자와 용어가 중요한 도메인에서는 BM25의 역할이 크다.

2. **Re-ranker는 "후보가 많을 때" 효과가 크다.** Hybrid Search로 20개 후보를 가져오고 Re-ranker가 5개를 선별하는 구조에서, Re-ranker는 노이즈를 제거하고 관련 청크를 상위로 올려 LLM 답변의 신뢰도를 높였다.

3. **년도 혼동은 메타데이터 태깅 + 프롬프트로 대부분 해결 가능하다.** 명시적 메타데이터 필터링 없이도, 프롬프트에 `[출처: {year}년 문서]`를 포함하는 것만으로 년도 검색 정확도 95~100%를 달성했다.

4. **자동 판정의 한계가 크다.** 키워드 매칭 기반 자동 판정은 실질적 정답을 오답으로 판정하는 경우가 많았다 (50% → 실질적 ~75%). LLM 기반 판정(LLM-as-Judge)이나 수동 판정 병행이 필요하다.

5. **근본적인 한계는 PDF 표 추출이다.** q02(2종 입원 10%), q14(2종 CT 15%)는 두 파이프라인 모두에서 실패했다. 이는 검색 기법의 문제가 아니라 PyPDFLoader가 표의 행/열 구조를 파괴하는 근본적 문제다. `pdfplumber`로 표를 마크다운으로 변환하면 해결 가능할 것이다.

## 9. 실습 전 가설 vs 실제 결과

### 가설 1: 년도 혼동이 가장 많이 발생할 질문 유형

**가설:** 두 년도에서 동일한 표(본인부담률 표)를 참조하는 질문에서 년도 혼동이 가장 많을 것이다.

**실제:** ❌ 예상과 달리 **년도 혼동은 거의 발생하지 않았다** (Basic: 100%, Advanced: 95.8%). 프롬프트에 년도를 명시하는 것만으로 LLM이 효과적으로 년도를 구분했다.

### 가설 2: Hybrid Search가 가장 효과를 볼 질문 유형

**가설:** 특정 의료 용어("추나요법", "KTAS")가 포함된 질문에서 BM25가 효과를 볼 것이다.

**실제:** ✅ **부분 일치.** BM25가 가장 효과를 본 것은 용어 자체보다 **표 데이터 검색**(q01, q03, q11)이었다. "1종", "외래", "1,000원" 같은 키워드 매칭이 깨진 표에서 정보를 찾는 데 결정적이었다.

### 가설 3: Re-ranking이 악화시킬 수 있는 경우

**가설:** Re-ranking이 키워드만 겹치는 무관한 청크를 밀어낼 수도 있지만, 오히려 관련 청크를 밀어내는 경우도 있을 것이다.

**실제:** ❌ **악화 사례 없음.** Re-ranking은 모든 문항에서 동일하거나 개선된 결과를 보였다. Cohere rerank-v3.5의 다국어 이해 능력이 한국어 의료 용어에도 효과적으로 작동한 것으로 보인다.

### 가설 4: 메타데이터 필터링의 년도 혼동 해결 효과

**가설:** 메타데이터 필터링이 년도 혼동을 크게 줄일 것이다.

**실제:** 메타데이터 필터링을 적용하지 않았지만, 프롬프트 기반 년도 표시만으로 95~100% 정확도를 달성했다. 메타데이터 필터링은 더 대규모 다년도 데이터에서 필요할 것으로 예상된다.
