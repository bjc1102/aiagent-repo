# 4주차 이론 과제: Advanced RAG — Hybrid Search, Re-ranking, 메타데이터 필터링

## 이론 조사 항목

### 1. Hybrid Search란?

**정의**: 벡터 검색(Semantic Search)과 키워드 검색(BM25)을 결합하여 두 방식의 장점을 모두 활용하는 검색 기법입니다.

**벡터 검색만으로 부족한 이유**

벡터 검색은 의미적 유사도를 기반으로 하므로, 아래 상황에서 한계가 있습니다:
- **정확한 키워드 매칭 실패**: "KTAS 4등급", "이상지질혈증" 같은 도메인 고유 명사·약어를 임베딩 공간에서 유사하게 처리하지 못할 수 있음
- **숫자·코드 검색 취약**: "제5조 2항", "100분의 5" 같은 표현은 의미보다 정확한 문자열 일치가 중요
- **도메인 희귀 용어**: 학습 데이터에 드물게 등장한 의료급여 전문 용어는 임베딩 품질이 낮을 수 있음

**BM25 알고리즘 핵심 원리**

BM25(Best Match 25)는 TF-IDF를 개선한 확률적 정보 검색 알고리즘입니다:
- **TF(Term Frequency)**: 문서 내 용어 출현 빈도에 비례해 점수 부여. 단, 반복이 많아도 점수가 무한히 증가하지 않도록 포화 함수(saturation) 적용 (파라미터 k1)
- **IDF(Inverse Document Frequency)**: 전체 문서 중 해당 용어가 등장하는 문서가 적을수록 높은 가중치 부여 — 희귀 용어일수록 중요
- **문서 길이 정규화**: 평균 문서 길이 대비 현재 문서의 상대 길이로 점수 보정 (파라미터 b). 긴 문서가 단순히 용어를 많이 포함한다는 이유로 유리해지는 것을 방지

| 검색 방식 | 장점 | 단점 |
|----------|------|------|
| **벡터 검색 (Semantic)** | 의미적 유사성 포착, 동의어·패러프레이징 처리, 다국어 지원 | 정확한 키워드 매칭 취약, 도메인 희귀 용어 처리 불안정, 계산 비용 높음 |
| **BM25 (Keyword)** | 정확한 키워드 매칭, 빠른 검색 속도, 결과 해석 용이 | 의미적 유사성 포착 불가, 동의어·문맥 이해 부재 |
| **Hybrid (결합)** | 두 방식의 약점을 상호 보완, 재현율(Recall)과 정밀도(Precision) 동시 향상 | 가중치 튜닝 필요, 구현 복잡도 증가, 두 시스템 유지 비용 발생 |

**결합 방식**

| 방식 | 설명 |
|------|------|
| **가중치 기반 합산** | `α × 벡터점수 + (1-α) × BM25점수` — 점수 스케일을 맞춘 뒤 가중 합산 (LangChain `EnsembleRetriever`의 `weights` 파라미터) |
| **Reciprocal Rank Fusion (RRF)** | 각 검색 결과의 순위를 `1/(k + rank)` 로 변환 후 합산 — 점수 정규화 불필요, 스케일 차이 영향 없음 |

### 2. Re-ranking이란?

**정의**: Retrieval 단계에서 얻은 후보 문서들을 더 정교한 모델로 재평가하여 순서를 재조정(Re-rank)하는 Post-Retrieval 기법입니다.

**Cross-encoder vs Bi-encoder 비교**

| 구분 | Bi-encoder | Cross-encoder |
|------|-----------|---------------|
| **입력 방식** | 질문과 문서를 **각각** 인코딩하여 임베딩 벡터 생성 후 코사인 유사도 계산 | 질문과 문서를 **함께** 입력으로 받아 단일 패스로 관련성 점수 출력 |
| **속도** | 빠름 — 문서 임베딩 사전 계산 가능 | 느림 — 쿼리-문서 쌍마다 별도 추론 필요 |
| **정확도** | 상대적으로 낮음 — 두 텍스트를 독립 인코딩하므로 교차 어텐션 없음 | 상대적으로 높음 — 두 텍스트 간 교차 어텐션으로 세밀한 관련성 파악 |
| **사용 위치** | 대규모 코퍼스 초기 검색 (Retrieval 단계) | 소수 후보 문서 재순위화 (Re-ranking 단계) |

**Cross-encoder를 검색 단계가 아닌 Re-ranking 단계에서 사용하는 이유**

- Cross-encoder는 질문과 문서를 쌍(pair)으로 처리하므로, N개 문서 전체에 적용 시 N번의 추론이 필요 — 수십만 개 문서에 실시간 적용 불가
- 따라서 **Bi-encoder로 후보 k개를 빠르게 선별(속도 우선)** → **Cross-encoder로 k개 중 최종 순위 결정(정확도 우선)** 의 Two-stage 패턴으로 속도와 정확도를 모두 확보

**Two-stage Retrieval 패턴**

```
전체 문서 (수만 건)
      │
      ▼  [Stage 1] Bi-encoder (벡터 검색 + BM25 Hybrid)
  후보 문서 k개 (예: 20개)
      │
      ▼  [Stage 2] Cross-encoder Re-ranking
  최종 상위 n개 (예: 3개)
      │
      ▼  LLM 답변 생성
```

**Cohere Rerank의 동작 방식과 장점**

- Cohere Rerank API는 Cross-encoder 기반의 상용 Re-ranking 모델로, 질문-문서 쌍의 관련성 점수(0~1)를 반환
- `rerank-v3.5` 모델은 다국어(한국어 포함)를 지원하며, 별도 모델 학습 없이 API 호출만으로 고품질 Re-ranking 적용 가능
- 직접 Cross-encoder를 운영하는 것 대비 인프라 비용 절감, 빠른 적용 가능

### 3. 메타데이터 필터링과 컨텍스트 압축

| 기법 | 한 줄 정의 | 필요성 | 구현 방식 예시 |
|------|----------|--------|--------------|
| **메타데이터 필터링** | 청크에 포함된 메타데이터(출처, 날짜, 카테고리 등)를 기반으로 검색 범위를 사전에 좁히는 기법 | 다년도 문서를 동시에 인덱싱한 경우, 연도 필터 없이 "2026년 장기지속형 주사제 본인부담률"을 검색하면 2025년 문서(5%)와 2026년 문서(2%)가 혼재 → LLM이 잘못된 연도의 수치를 답변할 위험 | FAISS: `search_kwargs={"k": 10, "filter": {"source_year": "2026"}}` |
| **Contextual Compression** | 검색된 청크에서 질문과 관련 있는 부분만 추출·압축하여 LLM에 전달하는 기법 | 없으면 청크 전체(수백 토큰)를 LLM에 그대로 전달 → 관련 없는 내용이 섞여 답변 정확도 저하, 토큰 비용 낭비 | LangChain `ContextualCompressionRetriever(base_compressor=CohereRerank(...), base_retriever=ensemble_retriever)` |

## 실습 과제 예측

### 1. 2025년과 2026년 문서를 동시에 인덱싱할 때 혼동이 가장 많이 발생할 질문 유형

**예측**: 연도별로 수치가 다른 본인부담률 관련 질문에서 가장 많은 혼동이 발생할 것입니다.

예를 들어 "장기지속형 주사제의 본인부담률"처럼 연도를 명시하지 않거나 자연어 속에 은닉된 질문은, 벡터 검색이 두 연도의 유사 청크를 모두 가져오면서 LLM이 2025년(5%)과 2026년(2%) 수치를 혼동할 가능성이 높습니다. 특히 숫자 차이가 크고 법령 개정으로 인한 변경 사항에 해당하는 항목들이 이에 해당합니다.

### 2. Hybrid Search가 가장 효과를 볼 질문 유형

**예측**: 정확한 도메인 고유 용어를 포함하는 질문에서 가장 큰 효과를 보일 것입니다.

예) "KTAS 4등급 응급의료관리료", "이상지질혈증 면제", "복잡추나" 등. 벡터 검색만으로는 이런 의료·행정 전문 용어를 의미적 유사도 기반으로 정확히 찾기 어렵지만, BM25가 정확한 토큰 매칭으로 보완합니다. 반면 일반적인 표현("진료비 얼마")은 벡터 검색이 이미 충분히 처리할 수 있어 Hybrid의 추가 이점이 상대적으로 작을 것입니다.

### 3. Re-ranking이 결과를 개선할지, 오히려 악화시킬 경우 예측

**개선이 기대되는 경우**: 벡터 검색과 BM25가 관련 청크를 후보로 선별했지만 순위가 잘못된 경우 — 예를 들어 연도가 섞인 20개 후보 중 정확한 연도의 청크가 하위 순위에 있을 때, Cross-encoder가 질문 전체 문맥과 가장 잘 매칭되는 청크를 상위로 올려줍니다.

**악화될 수 있는 경우**: 검색 후보 자체가 이미 질문과 무관한 청크만 포함된 경우 — Re-ranking은 후보를 새로 추가하지 않으므로, 20개 후보 모두 관련 없다면 상위 3개를 골라도 정답 청크가 없습니다. 또한 Re-ranking의 관련성 판단이 특정 도메인에서 예상과 다르게 동작하면 올바른 순위의 청크를 오히려 하위로 내릴 가능성도 있습니다.

### 4. 메타데이터 필터링이 혼동 문제를 해결할 수 있는 정도 예측

**효과적인 경우**: 질문에 연도가 명시되어 있거나 자연어 단서로 특정 연도를 감지할 수 있을 때 (예: "올해(2026년)", "2025년 규정"). `re.search(r"(2025|2026)", question)` 수준의 단순 파싱으로도 해당 연도 문서만 필터링하면 연도 혼동 문제의 대부분을 해결할 수 있을 것입니다.

**한계가 있는 경우**: ① 질문에 연도가 전혀 언급되지 않은 경우 → 필터 적용 불가, 전체 검색 폴백 ② 두 연도를 비교하는 질문 (예: "2025년과 2026년을 비교") → 단일 연도 필터 부적합, 전체 검색 필요 ③ 메타데이터가 올바르게 태깅되지 않은 청크 → 필터를 통과하지 못해 오히려 검색 실패

| 검색 실패로 인한 오답 | 4건 (q02, q06, q07, q11) |
| 주요 혼동 패턴 | "정보를 찾을 수 없습니다" 응답 — 연도 혼동이 아닌 top-K 청크 커버리지 부족 |

**2주차 Context LLM vs Basic RAG 비교**

| 방식 | 정답률 | 비고 |
|------|--------|------|
| Context LLM (전체 문서 in system prompt) | **100% (20/20)** | gemini-2.5-flash, PDF 전문을 COPAYMENT_REFERENCE로 내장 |
| 4주차 Basic RAG (벡터 Top-3) | **80% (16/20)** | 동일 LLM, 동일 golden dataset |

> Context LLM 대비 20%p 정확도 저하는 top-K=3의 낮은 재현율 때문. 2025/2026 양 연도 청크(총 26개) 중 3개만 검색하므로 관련 청크를 포함하지 못하는 경우 발생.
