# 0. 프롬프트 기법 4가지 이론 정리

## Zero-Shot Prompting (제로샷 프롬프팅)

### 한 줄 정리

제로샷 프롬프팅은 LLM에게 별도의 예시나 사전 학습 데이터 없이 바로 작업을 수행하도록 지시하는 기법입니다.

### 핵심 원리

별도의 예시 없이, 해당 모델이 이미 데이터를 통해 학습한 지식을 바탕으로 결과물을 추론하여 도출하는 방식입니다.

- LLM의 AR(autoregressive) 특성상 초기 생성되는 토큰들이 맥락을 결정하게 되기에, 질문이 모호하거나 학습 데이터가 적다면 맥락을 이탈하거나 환각(Hallucination)을 생성하는 빈도가 높아질 수 있습니다.

### 사용처

- **효과적인 경우:** 범용적인 질의응답, 단순 텍스트 처리 등 지시어 정도로 충분한 작업
- **효과가 적은 경우:** 도메인 특화 지식이 필요한 경우, 엄격한 출력 형식 유지가 필요한 경우, 다단계 추론이 필요한 경우

---

## Few-Shot Prompting (퓨샷 프롬프팅)

### 한 줄 정리

퓨샷 프롬프팅은 원하는 결과에 대한 예시를 먼저 주고, LLM이 추론하는 데 필요한 패턴과 형식을 컨텍스트로 학습시키는 방법입니다.

### 핵심 원리

모델을 별도로 학습시키지 않고, 입력된 문맥만으로 모델의 행동을 변화시키는 문맥 내 학습(In-Context Learning)을 핵심으로 합니다.

- **확률 분포에서의 강력한 조건부 역할:** 제로샷이 거대한 확률 지표에서 자유롭게 추론한다면, 퓨샷은 데모라는 이정표를 미리 세워 수많은 패턴 중 사용자가 원하는 특정 패턴의 확률값을 높입니다.
- **레이블 공간과 형식:** 답변으로 나올 수 있는 후보군을 모델에게 미리 인지시키고, 예시의 정답이 틀려도 일정 형식을 유지시키는 것 자체가 모델이 답변 궤도에 머물도록 도와주는 강력한 가이드가 됩니다.

### 사용처

- **효과적인 경우:** 정형화된 데이터 추출(JSON 등), 복잡한 분류 작업
- **효과가 적은 경우:** 다단계 논리 추론이 필요한 경우, 맥락 창(Context Window)이 좁은 모델에서 예시가 길어지는 경우

---

## Chain-of-Thought (CoT, 생각의 사슬)

### 한 줄 정리

CoT는 복잡한 문제를 한 번에 해결하지 않고, 논리적인 중간 추론 단계를 순차적으로 생성하여 정답의 정확도를 높이는 기법입니다.

### 핵심 원리

"결과(Result)"가 아닌 "사고의 과정(Reasoning)"을 퓨샷 데모로 제공하여, 모델이 스스로 정답에 도달하기 위한 중간 논리 단계를 생성하도록 유도하는 기법입니다.

- **추론을 통한 계산 공간 확보:** CoT는 중간 추론 과정을 출력하게 해서 생각할 시간과 공간을 줍니다. 추론한 내용들이 다시 입력으로 들어가면서(AR 효과), 최종 정답 토큰이 생성될 때의 확률 분포를 매우 정교하게 가이드할 수 있게 됩니다.
- **Zero-shot CoT:** "단계별로 생각해보자"라는 문구만으로도 성능이 올라갈 수 있습니다. 모델이 학습해둔 논리적 전개 패턴을 트리거하여, 단답형이 아닌 설명형/단계형으로 강하게 전이시켜 정답에 도달하기 위한 유의미한 결과물 도출에 도움을 줍니다.
- **Auto-CoT:** 수작업으로 CoT 예시를 만드는 번거로움을 해결하고, 예시의 편향성을 줄일 수 있습니다. 질문 클러스터링과 휴리스틱 필터링을 통해 양질의 논리 경로를 예시로 제공합니다.

### 사용처

- **효과적인 경우:** 복잡한 논리 추론(수학, 물리 문제 풀이), 코드 아키텍처 설계(디자인 패턴 적용, 리팩토링)
- **효과가 적은 경우:** 실시간 응답이 필수인 경우(출력 토큰 증가로 인한 지연), 단순 사실 확인

---

## Self-Consistency (자기 일관성)

### 한 줄 정리

하나의 고정된 추론 경로만 따라가지 않고, 여러 개의 다양한 추론 경로를 독립적으로 생성하고, 가장 많이 도출된 공통 답변을 최종 정답으로 선택하는 기법입니다.

### 핵심 원리

단일 추론 경로에서 발생할 수 있는 우발적 오류(Greedy Decoding의 한계)를 극복하기 위해, 여러 번의 '생각의 사슬(CoT)'을 시도하여 결과의 신뢰도와 일관성을 확보합니다.

- **탐욕적 디코딩(Greedy Decoding) 대체:** 모델이 가장 확률이 높은 토큰만 선택하는 방식 대신, 다양한 후보군을 샘플링합니다.
- **추론 경로의 다양성 확보:** 동일 질문에 대해 조금씩 다른 논리 전개를 하도록 유도하여, 틀린 길로 빠지는 것을 방지합니다.

### 사용처

- **효과적인 경우:** 산술 추론 등 계산 실수가 일어나면 안 되는 정밀한 작업
- **효과가 적은 경우:** 비용과 속도가 중요한 경우, 창의적인 결과 도출이 필요한 경우

## 예측

### 1. Zero-shot(Step 1)에서 정답률이 어느 정도일지 예측하고 이유를 설명하세요

전체 평균 정답률 **70% 이상** 예상됩니다.

이미지의 텍스트 추출(OCR) 테스트 결과가 높아서 이미지 텍스트 추출 정상 작동을 가정하게 되었으며, 단순 정보 추출에서는 완벽에 가까운 성능을 보이나 복합 추론으로 갈수록 정답률이 점진적으로 하락할 것으로 분석됩니다.

| **난이도** | **정답률(예측)** | **핵심 특징 및 판단 근거** |
|---|---|---|
| **Easy (단순 추출형)** | **90% ~ 100%** | 이미지의 표 내용을 거의 완벽하게 읽어낼 수 있으므로, 키워드 매칭만으로 정답 도출이 가능한 단계입니다. |
| **Medium (복합 조건 및 비교형)** | **70% ~ 80%** | 단순 추출을 넘어 여러 필터(예: 2종 + 디스크 외 + 복잡추나)를 동시에 적용해야 하므로, 단계별 해석 오류가 발생할 가능성이 존재합니다. |
| **Hard (추론 및 연산 적용형)** | **약 50%** | 정확한 정보 추출에 더해 '복합 계산 로직'이 추가됩니다. 수치 연산과 추론 과정이 결합되는 지점에서 모델의 할루시네이션(환각) 위험이 가장 높습니다. |

### 2. 어떤 난이도(easy/medium/hard)에서 가장 큰 개선이 일어날지 예측하세요

| **난이도** | **개선 예상** |
|---|---|
| **Easy (단순 추출형)** | 유의미한 개선 결과는 없을 것으로 예상됩니다. |
| **Medium (복합 조건 및 비교형)** | 많은 예시 데이터 없이도, 비교형 조건 추가를 잘 하는 것으로도 개선을 빠르게 일으킬 수 있을 것으로 예상되기에, **Medium 난이도에서 가장 큰 개선**이 일어날 것으로 예상됩니다. |
| **Hard (추론 및 연산 적용형)** | 여전히 예시 샘플이나 프롬프팅의 개선이 필요하기에, 개선은 되겠지만 Medium보다는 덜 개선될 것으로 예상됩니다. |

### 3. 4가지 기법 중 가장 큰 정답률 점프가 어디서 일어날지 예측하세요

필요한 조건들을 확인하여 표를 읽고 결과를 합산하거나 선택하여 도출하는 방식이 중요하기에, 단계적으로 수행할 수 있게 도와주는 **CoT 프롬프팅**이 가장 큰 개선효과가 있을 것으로 예상됩니다.

# 2. 각 Step별 프롬프트 전문

초기 구현에서는 Gemini 비전으로 원본 이미지(`week-2/image/image.png`)를 직접 읽어 참조 데이터를 추출하는 OCR 기반 방식으로 실험을 진행했습니다. 그런데 이 초기 zero-shot 실험에서도 이미 정확도가 충분히 높게 나왔기 때문에, 이후에는 참고 자료 입력 방식 자체를 계속 비교하기보다는 프롬프트 기법별 차이를 비교하는 쪽에 더 의미가 있다고 판단했습니다. 또한 반복 실험을 여러 번 수행하는 과정에서 이미지 OCR 호출이 매번 추가 토큰을 사용하므로, 이후 테스트 단계에서는 한 번 추출해 저장한 `week-2/DChanHong/extracted-reference.md`를 직접 읽어오도록 변경했습니다. 즉, 이후 실험은 이미지 재인식 대신 정리된 참조 마크다운을 재사용하는 방식으로 진행했습니다.

실제 system prompt는 아래 공통 prefix 뒤에 Step별 본문이 붙는 형태로 구성했습니다.

```text
## 참조 데이터
<extracted-reference.md 내용 전체>

---
<Step별 프롬프트 본문>
```

공통 출력 형식 지시는 모든 Step에서 동일하게 사용했습니다.

```json
{
  "answer": "최종 답만 간결하게 (예: 5%, 무료, 210,000원)",
  "reason": "참조 데이터의 어느 규칙을 적용했는지 짧게 요약"
}
```

## Step 1. Zero-shot Prompt

```text
당신은 의료급여 본인부담률 질문에 답하는 보조 도구입니다.
제공된 참조 데이터만을 근거로 질문에 맞는 본인부담률(또는 본인부담금)을 결정하세요.

반드시 아래 JSON 형식으로만 응답하세요. JSON 바깥에 설명 문장을 붙이지 마세요.
{
  "answer": "최종 답만 간결하게 (예: 5%, 무료, 210,000원)",
  "reason": "참조 데이터의 어느 규칙을 적용했는지 짧게 요약"
}
```

## Step 2. Few-shot Prompt

```text
당신은 의료급여 본인부담률 질문에 답하는 보조 도구입니다.
Few-shot 예시를 참고해 질문 조건에 맞는 규칙을 적용하세요.
정답 의미에 포함되는 조건(예: 병원급 이상, 의료기관 차수)은 생략하지 마세요.

## 예시
Q: 2종 수급권자의 조현병 외 정신질환 환자가 외래 진료를 받으면 본인부담률은 얼마인가요?
A:
{
  "answer": "병원급 이상 10%",
  "reason": "정신질환 외래진료에서 조현병 외 정신질환은 병원급 이상 10%입니다."
}

Q: 65세 이상 2종 수급권자가 임플란트를 할 때 본인부담 보상제나 상한제가 적용되나요?
A:
{
  "answer": "해당되지 않음",
  "reason": "65세 이상 틀니 및 치과 임플란트 항목은 본인부담 보상제·상한제에 해당되지 않습니다."
}

Q: 65세 이상 1종 수급권자가 틀니(시술비 1,500,000원)와 임플란트(시술비 1,000,000원)를 동시에 하면 각각의 본인부담금은 얼마인가요?
A:
{
  "answer": "틀니 75,000원, 임플란트 100,000원",
  "reason": "1종은 틀니 5%, 임플란트 10%이므로 각각 1,500,000×0.05=75,000원, 1,000,000×0.10=100,000원입니다."
}

반드시 아래 JSON 형식으로만 응답하세요. JSON 바깥에 설명 문장을 붙이지 마세요.
{
  "answer": "최종 답만 간결하게 (예: 5%, 무료, 210,000원)",
  "reason": "참조 데이터의 어느 규칙을 적용했는지 짧게 요약"
}
```

## Step 3. Chain of Thought Prompt

```text
당신은 의료급여 본인부담률 질문에 답하는 보조 도구입니다.
최우선 목표는 `reason`이 아니라 `answer`를 정확한 형식으로 맞히는 것입니다.
`reason`은 짧게 쓰고, 내부 판단 결과를 바탕으로 `answer`를 정확히 확정하세요.

`reason`에는 아래만 2~3단계로 아주 짧게 적으세요.
1) 질문에서 핵심 조건 추출
2) 적용한 섹션/행과 비율 확인
3) 계산 문제면 계산식 1줄

추가 규칙:
- `CT, MRI, PET 등`, `추나요법`, `틀니/임플란트`처럼 항목명이 직접 나오면 해당 전용 표를 우선합니다.
- 일반 규칙과 특수 검사/시술 규칙이 동시에 보이면, 더 구체적인 특수 규칙을 선택합니다.
- 참조 데이터에 없는 비율을 임의로 만들지 말고, 다른 행의 수치를 그대로 옮겨 추측하지 마세요.
- `answer`는 가능한 한 참조 데이터의 표현을 그대로 사용하세요.
- 비율 답변에서 조건이 답 의미에 포함되면 생략하지 마세요. 예: `병원급 이상 10%`
- 예외 답변은 `적용되지 않습니다`처럼 바꾸지 말고 `해당되지 않음`처럼 참조 표현에 맞추세요.
- 복수 금액 답변은 콜론 없이 `틀니 75,000원, 임플란트 100,000원` 형식을 사용하세요.
- 금액 답변은 계산 후 최종 금액만 쓰고, 불필요한 설명이나 단위를 추가로 붙이지 마세요.
- `reason`은 짧게 유지하고, `answer` 표현을 바꾸는 근거로 사용하지 마세요.

`answer`에는 결과만 간결히 적습니다.

반드시 아래 JSON 형식으로만 응답하세요. JSON 바깥에 설명 문장을 붙이지 마세요.
{
  "answer": "최종 답만 간결하게 (예: 5%, 무료, 210,000원)",
  "reason": "참조 데이터의 어느 규칙을 적용했는지 짧게 요약"
}
```

## Step 4. Self-Consistency Prompt

Self-Consistency는 Step 3의 CoT 프롬프트를 그대로 사용하되, 아래 지시를 추가로 붙였습니다.

```text
당신은 Self-Consistency용 의료급여 본인부담률 답변 보조 도구입니다.
각 생성은 `_COT_INSTRUCTION`의 규칙을 따르되, 최우선 목표는 항상 `answer`를 정확한 형식으로 맞히는 것입니다.

Self-Consistency 규칙:
- 같은 질문을 여러 번 풀더라도 `answer`의 최종 표현은 참조 데이터 기준으로 최대한 동일하게 유지하세요.
- 다양성이 필요할 때는 `reason`의 서술 방식만 조금 달라질 수 있고, `answer` 표현은 흔들지 마세요.
- 다수결 대상은 `answer`이므로, 동의어/패러프레이즈로 `answer`를 바꾸지 마세요.
- 비율/금액/예외 표현 형식은 `_COT_INSTRUCTION`의 답안 형식 규칙을 그대로 따르세요.
- temperature가 높아도 참조 데이터에 없는 비율이나 금액을 새로 만들지 마세요.

반드시 아래 JSON 형식으로만 응답하세요. JSON 바깥에 설명 문장을 붙이지 마세요.
{
  "answer": "최종 답만 간결하게 (예: 5%, 무료, 210,000원)",
  "reason": "참조 데이터의 어느 규칙을 적용했는지 짧게 요약"
}
```

# 3. 결과 요약 테이블

아래 결과는 `extracted-reference.md`를 참조 데이터로 사용한 최종 재실험 기준입니다.

| Step | 프롬프트 기반 구성 | 평가 제외 | 정확도 | 평균 총 토큰/문항 | 평균 시간/문항 |
|---|---|---|---:|---:|---:|
| Zero-shot | 참조 데이터 + 직접 추론 + 구조화 출력 | 없음 | 28/30 (`0.9333`) | `1458.13` | `2314.14ms` |
| Few-shot | Zero-shot + 예시 3개(`q12`, `q15`, `q26`) | `q12,q15,q26` | 27/27 (`1.0`) | `1816.2` | `2180.66ms` |
| CoT | answer 중심 CoT + 구조화 출력 | 없음 | 30/30 (`1.0`) | `1993.3` | `3158.95ms` |
| Self-Consistency | CoT + temperature sweep(`0.3,0.5,0.7`) + answer 다수결 | 없음 | 30/30 (`1.0`) | `6741.93` | `8021.06ms` |

## 해석

- `Zero-shot`은 가장 단순한 구성으로도 `0.9333`의 정확도를 보였습니다.
- `Few-shot`은 예시 3개를 추가하고 해당 문항을 평가에서 제외한 조건에서 `1.0`을 기록했습니다.
- `CoT`는 초반 프롬프트에서는 오히려 성능이 하락했지만, `answer` 형식 고정 중심으로 수정한 뒤 `1.0`으로 개선되었습니다.
- `Self-Consistency`도 `1.0`을 기록했지만, temperature를 3개로 늘려 다수결을 수행하므로 토큰과 시간 비용이 가장 크게 증가했습니다.

# 5. Step 1 → Step 4까지 정답률 개선 과정과 인사이트

이번 실험에서는 `Zero-shot -> Few-shot -> CoT -> Self-Consistency` 순서로 프롬프트를 점진적으로 확장하며 성능 변화를 확인했습니다. 전체적으로 보면, 무조건 복잡한 기법을 먼저 적용하기보다 가장 단순한 방법부터 시작해 필요한 경우에만 다음 기법을 추가하는 접근이 더 합리적이라는 점을 확인할 수 있었습니다.

## Step 1. Zero-shot

`Zero-shot`만으로도 예상보다 높은 정확도를 확인할 수 있었습니다. 특히 참조 데이터 표를 정확하게 읽어오고, 구조가 잘 정리된 상태라면 별도의 예시 없이도 충분히 좋은 성능을 낼 수 있었습니다. 즉, 프롬프트를 복잡하게 만들기 전에 먼저 참조 데이터 품질과 입력 구조를 잘 정리하는 것이 매우 중요하다는 점을 확인했습니다.

## Step 2. Few-shot

`Few-shot`에서는 모델이 자주 실패하거나 형식을 흔들리는 지점을 미리 파악한 뒤, 그에 맞는 예시를 system prompt에 추가했습니다. 그 결과 단순히 정답을 알려주는 것이 아니라, 모델이 어떤 방식으로 답을 써야 하는지에 대한 기준을 더 명확히 줄 수 있었습니다. 결국 완성도를 높이기 위해서는 AI가 실패하는 패턴을 먼저 분석하고, 그 취약 지점을 예시로 보완하는 방식이 효과적이었습니다.

## Step 3. Chain of Thought

`CoT`는 분명 도움이 될 수 있는 기법이지만, 이번 실험에서는 초기 프롬프트를 잘못 설계했을 때 오히려 정답률이 떨어지는 현상도 확인했습니다. 특히 `reason`을 과도하게 강조하면 `answer` 형식이 흔들리거나 불필요한 표현 변화가 생겨 exact match 기준에서는 손해가 날 수 있었습니다. 따라서 CoT는 단순히 단계적 추론을 많이 쓰게 하는 것이 아니라, 어떤 방향으로 추론을 유도할지와 최종 답 형식을 어떻게 고정할지를 함께 설계하는 것이 중요했습니다.

## Step 4. Self-Consistency

`Self-Consistency`는 정확도를 높일 수 있는 좋은 방법이지만, 여러 번 생성한 뒤 다수결을 수행해야 하므로 토큰 비용과 실행 시간이 크게 증가했습니다. 따라서 정확도가 가장 중요한 상황에서는 유용할 수 있지만, 비용이나 응답 속도가 중요한 서비스 환경에서는 항상 적합한 선택이라고 보기는 어렵습니다. 결국 이 방법은 성능 향상 가능성과 비용 증가를 함께 고려해 선택해야 하는 기법이라고 판단했습니다.

## 종합

전체적으로 보면, 필요한 경우 고급 프롬프팅 기법을 적용할 수는 있지만, 처음부터 복잡한 방법을 쓰기보다는 `Zero-shot`부터 시작해 결과를 확인하고, 그 다음에 `Few-shot`, `CoT`, `Self-Consistency`처럼 점진적으로 확장하며 비교하는 방향이 가장 현실적이었습니다. 즉, 프롬프팅 기법 선택은 “더 복잡한 기법이 항상 더 좋다”가 아니라, 현재 문제와 비용, 시간, 정확도 요구사항에 맞게 단계적으로 검증하면서 결정하는 것이 더 적절하다고 보았습니다.

# 6. 가설 전후 비교

실험 전 세운 예측과 실제 결과를 비교합니다.

## 가설 1. Zero-shot 정답률 예측

| 난이도 | 예측 정답률 | 실제 결과 | 일치 여부 |
|---|---|---|---|
| Easy (단순 추출형) | 90% ~ 100% | 28/30 전체 정확도 `0.9333` — Easy 문항 대부분 정답 | **일치** |
| Medium (복합 조건 및 비교형) | 70% ~ 80% | 대부분 정답, 예상보다 높음 | **예측보다 높음** |
| Hard (추론 및 연산 적용형) | 약 50% | 30문항 중 2문항만 오답 — Hard에서도 대부분 정답 | **예측보다 훨씬 높음** |

**분석:** 전체 평균 70% 이상이라는 예측은 맞았지만, 실제로는 `93.3%`로 예측을 크게 상회했습니다. 가장 큰 차이는 Hard 난이도에서 발생했는데, `extracted-reference.md`로 참조 데이터를 정확하게 전달한 것이 모델의 추론 정확도를 끌어올린 핵심 요인이었습니다. 즉, 참조 데이터 품질이 높으면 Zero-shot만으로도 복합 추론 문항까지 커버할 수 있다는 점이 확인되었습니다.

## 가설 2. 가장 큰 개선이 일어나는 난이도

| 난이도 | 예측 | 실제 결과 |
|---|---|---|
| Easy | 유의미한 개선 없음 | Zero-shot부터 거의 전부 정답 → 개선 여지 없음 |
| **Medium** | **가장 큰 개선** 예상 | Zero-shot에서 이미 대부분 정답이라 추가 개선 폭은 제한적 |
| Hard | Medium보다 덜 개선 | 2문항 오답 → Few-shot/CoT로 해소, 실질적 개선은 Hard에서 발생 |

**분석:** Medium에서 가장 큰 개선이 일어날 것이라는 예측과 달리, 실제로는 **Hard 문항에서 실질적 개선**이 발생했습니다. Zero-shot 단계에서 Medium이 이미 높은 정답률을 보였기 때문에, 개선 여지가 남아있던 것은 Hard 문항이었습니다. Few-shot 예시로 취약 패턴을 보완하고 CoT로 추론 단계를 유도한 것이 Hard 문항의 오답 2개를 해소하는 데 직접적으로 기여했습니다.

## 가설 3. 가장 큰 정답률 점프가 일어나는 기법

| 항목 | 예측 | 실제 결과 |
|---|---|---|
| 가장 큰 점프 기법 | **CoT** | **Few-shot** (`0.9333` → `1.0`) |
| CoT 실제 기여 | 단계별 추론으로 큰 개선 기대 | 초기 CoT는 오히려 정답률 하락, 프롬프트 재설계 후 `1.0` 달성 |
| Self-Consistency | - | 정확도 `1.0` 유지, 토큰·시간 비용만 증가 |

**분석:** CoT에서 가장 큰 점프가 일어날 것이라는 예측은 **빗나갔습니다**. 실제로는 `Few-shot` 단계에서 `0.9333 → 1.0`으로 정답률이 올라간 것이 가장 뚜렷한 점프였습니다. CoT는 오히려 초기 프롬프트 설계에서 `reason`을 과도하게 강조해 `answer` 형식이 흔들리는 부작용을 겪었고, 프롬프트를 재설계한 뒤에야 `1.0`을 달성했습니다.

이 결과는 "단계적 추론이 항상 최선"이라는 가정이 모든 상황에 성립하지 않음을 보여줍니다. 이번 과제처럼 **참조 데이터가 잘 정리되어 있고 exact match 기준으로 평가하는 환경**에서는, 복잡한 추론보다 정확한 출력 형식을 예시로 보여주는 Few-shot이 더 직접적인 효과를 발휘했습니다.
