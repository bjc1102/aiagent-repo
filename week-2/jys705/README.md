# 2주차 과제: 프롬프트 엔지니어링으로 LLM 개선하기

## Part 1. 이론 과제: 프롬프트 엔지니어링 기법 조사

### 1. 프롬프트 기법 4가지 정리
* **Zero-shot Prompting**
    * **한 줄 정의**: 모델에게 예시나 추가적인 힌트 없이 지시사항만 주고 바로 답변을 요구하는 기법.
    * **핵심 원리**: LLM이 방대한 사전 학습 데이터를 바탕으로 이미 알고 있는 지식을 활용해 즉각적으로 대답함.
    * **장단점**: 구현이 매우 쉽고 단순한 태스크에 적합하지만, 복잡한 추론이나 특정 포맷을 요구하는 작업에서는 성능이 크게 떨어짐.
* **Few-shot Prompting**
    * **한 줄 정의**: 프롬프트 내에 질문과 정답 쌍으로 이루어진 예시(Example)를 몇 개 제공하여 모델의 답변 방향을 가이드하는 기법.
    * **핵심 원리**: In-Context Learning을 통해 모델이 예시의 패턴과 출력 형식(Format)을 모방하여 답변함.
    * **장단점**: 원하는 포맷(예: JSON, 특정 말투)을 강제하는 데 매우 탁월하지만, 복잡한 수학 연산이나 논리적 추론 능력을 직접적으로 올려주지는 못함.
* **Chain-of-Thought (CoT)**
    * **한 줄 정의**: 최종 답을 내리기 전에, 단계별로 추론하는 과정(Thinking process)을 먼저 작성하도록 강제하는 기법.
    * **핵심 원리**: LLM은 이전에 생성한 토큰을 기반으로 다음 토큰을 예측(Autoregressive)하므로, 스스로 내뱉은 논리 과정을 징검다리 삼아 복잡한 문제의 연산 능력을 극대화함.
    * **장단점**: 다중 조건이나 수학 문제 해결 능력이 비약적으로 상승하지만, 추론 과정이 길어져 출력 토큰 비용이 증가함.
* **Self-Consistency (자기 일관성)**
    * **한 줄 정의**: 동일한 질문을 여러 번 생성(Sampling)한 뒤, 가장 많이 나온 답(다수결)을 최종 정답으로 채택하는 기법.
    * **핵심 원리**: CoT를 사용할 때 다양한 추론 경로를 탐색하게 하고(Temperature 증가), 일시적인 환각(Hallucination)이나 계산 실수를 집단 지성으로 걸러냄.
    * **장단점**: 단일 호출보다 훨씬 견고하고 정확한 답을 얻을 수 있으나, API 호출 횟수와 대기 시간이 배수로 증가함.

### 2. 실습 전 가설(Hypothesis) 수립
1.  **Zero-shot 정답률 예측**: 표 데이터의 위상을 제대로 파악하지 못하고, 다중 조건 계산에 실패하여 약 **20~30%**의 낮은 정답률을 보일 것이다.
2.  **최대 개선 난이도**: 금액을 직접 계산해야 하는 **Hard 문항**에서 가장 극적인 정답률 상승이 일어날 것이다.
3.  **가장 큰 점프 구간 예측**: 모델 스스로 논리 트리를 타도록 강제하는 **Few-shot → CoT 적용 구간**에서 가장 큰 정답률 점프가 일어날 것이다.

## Part 2. 실습 과제: 의료급여 본인부담률 산정 태스크

### 1. 사용한 모델, SDK, 실행 환경
* **모델**: `gpt-4o-mini` (복잡한 추론의 가성비 및 다수결 반복 호출 테스트 목적)
* **SDK**: `openai` Python SDK / `pydantic` (Structured Outputs 적용)
* **실행 환경**: Python 3.10 / macOS

### 2. 각 Step별 프롬프트 전문 및 정답률

#### [Step 1] Zero-shot Baseline (정답률: 70.0% - 21/30)
```python
system_prompt = f"""아래는 의료급여 본인부담률 참조 데이터입니다. 
질문에 대해 정확한 본인부담률 또는 본인부담금을 답하세요. 답만 간결하게 작성하세요.

{copayment_reference}"""
```
* **결과**: 가설(20~30%)보다 기본 지능이 높아 많은 문제를 맞췄으나, '0%'를 '무료'로 쓰지 않거나, 콤마(,)와 '원'을 생략하는 등 **출력 포맷 불일치로 인한 억울한 오답**이 대거 발생함.

#### [Step 2] Few-shot Prompting (정답률: 76.7% - 23/30)
```python
system_prompt = f"""아래는 의료급여 본인부담률 참조 데이터입니다. 
질문에 대해 정확한 본인부담률 또는 본인부담금을 답하세요. 답만 간결하게 작성하세요.

[답변 작성 예시]
Q: 2종 수급권자인 생후 8개월 아기가 만성질환자이고 제2차의료급여기관에서 외래 진료를 받으면 본인부담률은?
A: {{"reason": "1세 미만 만성질환자 + 제2차의료급여기관 = 무료", "answer": "무료"}}

Q: 1종 수급권자가 디스크로 복잡추나를 받았고 치료비가 100,000원입니다. 본인부담금은?
A: {{"reason": "디스크 + 1종 복잡추나 = 30%. 100,000 * 0.3 = 30,000원", "answer": "30,000원"}}

Q: 65세 이상 2종 수급권자가 틀니(시술비 1,000,000원)와 임플란트(시술비 1,000,000원)를 동시에 하면?
A: {{"reason": "2종 틀니 15%, 임플란트 20%", "answer": "틀니 150,000원, 임플란트 200,000원"}}

{copayment_reference}"""
```
* **결과**: 포맷 오류(억울한 오답)는 예시 덕분에 완벽히 교정됨. 그러나 다중 조건을 교차 검증하지 못하고 나이 구간이나 질환을 잘못 매핑하는 논리적 한계는 여전히 존재함.

#### [Step 3] Chain-of-Thought (정답률: 66.7% - 20/30)
```python
system_prompt = f"""당신은 국민건강보험공단의 본인부담률 산정 전문가입니다. 
아래 의료급여 본인부담률 참조 데이터를 바탕으로 질문에 답하세요.

[추론 지침 - Chain of Thought]
최종 정답(answer)을 도출하기 전에 'reason' 필드에 반드시 다음 단계를 거쳐 생각하세요:
1단계: 수급자 종별(1종/2종) 파악
2단계: 환자의 연령 및 특이사항(임신, 질환 유무, 만성질환 등) 파악
3단계: 진료 항목(입원/외래, 치료 종류, 검사 종류) 및 의료기관 종별(1차/2차/3차) 파악
4단계: 참조 데이터에서 위 조건들에 완벽히 부합하는 본인부담률(%) 찾기 (일반 규칙보다 특수/예외 규칙 우선 적용)
5단계: 만약 진료비/치료비 금액이 주어졌다면, 찾은 비율을 곱하여 최종 본인부담금(원) 계산하기

{copayment_reference}"""
```
* **결과**: **조합을 고려하지 못함.** 논리력은 상승하여 어려운 문제(예외 조항 등)를 풀어냈으나, 예시(Few-shot)가 빠지자 출력 포맷이 완전히 무너져(예: '무료' 대신 '0%') 표면적인 정답률이 오히려 하락함. **"CoT와 Few-shot은 결합되어야만 시너지가 난다"**는 교훈을 얻음.

#### [Step 4] Self-Consistency (정답률: 80.0% - 24/30)
* **적용 기법**: Step 3의 CoT 지침 + Step 2의 Few-shot 예시 결합 + `temperature=0.7` 환경에서 **5회 반복 호출 후 다수결(Majority Vote)** 채택
* **결과**: 80.0%로 최고 정답률 달성. 일시적인 계산 실수나 엉뚱한 비율을 가져오는 환각을 다수결 투표가 방어해 냄.

### 3. 결과 요약 테이블

| 단계 (Step) | 적용 프롬프트 기법 | 정답률 | 주요 특징 및 변화 |
| :--- | :--- | :---: | :--- |
| **Step 1** | Zero-shot | 70.0% | 모델 지능은 높으나 포맷 붕괴로 억울한 오답 대거 발생 |
| **Step 2** | Few-shot | 76.7% | 포맷은 교정되었으나 다중 조건 논리 탐색 한계 |
| **Step 3** | CoT (Only) | 66.7% | 논리력은 올랐으나 폼이 무너져 점수 하락 (역설적 현상) |
| **Step 4** | **CoT + Few-shot + 다수결** | **80.0%** | 집단 지성으로 계산 실수 및 논리 오류 방어 (최고점) |

*(※ 추가 실험: 다수결 없이 `temperature=0` 환경에서 CoT+Few-shot만 결합했을 때 **83.3%** 달성. 룰 기반 태스크에서는 과도한 창의성(T=0.7)이 독이 될 수 있음을 확인함)*

### 4. 오답 분석 — 가장 어려웠던 문항 3건과 원인

겹겹이 방어막을 쳤음에도 LLM이 끝까지 틀린 대표적인 3가지 유형의 오답입니다.

1. **경계값(Boundary) 판별 실패 [q03]**
    * **현상**: "4세 아동"이 입원했을 때, 표의 `6세 미만(무료)` 구간을 찾지 못하고 `6세 이상~15세 이하(3%)`로 분류함.
    * **원인**: LLM은 수학적 크기 비교보다 텍스트 표면의 패턴 매칭에 의존함. 표에 '4세'라는 명시적 텍스트가 없자 엉뚱한 나이 구간을 가져오는 수학적 맹점을 보임.
2. **특수 예외 규칙 간과 (Attention Slips) [q19]**
    * **현상**: "고위험 임신부"의 진료비를 물었을 때, `고위험 임신부(5%)` 예외를 무시하고 `자연분만/제왕절개(무료)`라는 일반 임신부 규칙에 꽂혀 '무료'로 오답을 냄.
    * **원인**: 프롬프트 길이가 길어지자, LLM이 문맥을 위에서 아래로 읽다가 첫 번째 조건(임신부=무료)에 주의력(Attention)을 뺏겨 그 하단에 있는 예외 조항을 교차 검증하지 못함.
3. **Few-shot Bias (과적합) [q09]**
    * **현상**: "생후 8개월 아기"를 갑자기 "만성질환자"로 지어내어 오답 처리됨.
    * **원인**: 시스템 프롬프트에 넣은 예시(Few-shot)가 하필 '만성질환 아기'였음. 모델이 예시에 너무 과적합되어 실제 문제에 없는 병을 환각(Hallucination)으로 만들어냄.

### 5. Step 1 → Step 4 개선 과정과 최종 인사이트

이번 실습을 통해 프롬프트 엔지니어링이 어느정도로 가능한지, 그리고 한계까지 체험할 수 있었습니다.

1. **프롬프트 기법의 시너지**: CoT는 논리를, Few-shot은 형식을 보장합니다. 어느 하나만 단독으로 쓰일 때보다 둘이 결합될 때 비로소 강력한 에이전트가 됨을 Step 3의 실패를 통해 배웠습니다.
2. **Self-Consistency의 역설**: 창의성(`temperature=0.7`)을 부여한 다수결 방식은 브레인스토밍에는 좋지만, **엄격한 룰(Rule) 기반의 표 데이터 찾기에서는 오히려 창의적인 오답(환각)을 양산**하여 다수결의 정확도를 떨어뜨리는 부작용을 확인했습니다. (Temp=0 환경이 오히려 83.3%로 높았음)
3. **데이터 구조화의 중요성 (넥스트 스텝)**: LLM이 80%대에서 한계에 부딪힌 근본적 이유는 텍스트(Markdown)로 평면화된 표 데이터 때문입니다. 100% 정답률을 달성하기 위해서는 프롬프트 길이를 늘리는 것이 아니라, **참조 데이터 자체를 JSON이나 YAML 같은 트리(Tree) 구조로 변환하여 위계질서와 예외 규칙을 명확히 주입해야 한다**는 몰랐던 사실을 얻었습니다.
