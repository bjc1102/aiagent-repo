
## 1. 프롬프트 기법 4가지 정리

### ① Zero-shot Prompting (제로샷 프롬프팅)
* **한 줄 정의:** 모델에게 어떠한 예시나 사전 단서 없이 곧바로 질문이나 지시를 내리는 가장 기본적인 기법입니다.
* **핵심 원리 (Autoregressive 연결):** 이전 단어들을 기반으로 다음 단어를 예측하는 자기회귀(Autoregressive) 생성 방식에서, 오직 '사전 학습된 가중치'에만 의존하여 정답 토큰을 예측합니다. 프롬프트 내에 컨텍스트가 부족하므로 모델의 기본 역량에 전적으로 의존합니다.
* **사용 예시:** `"이 Jenkins 파이프라인 스크립트에서 에러가 나는 이유를 찾아줘. [스크립트 코드]"`
* **언제 쓰면 좋은지 / 효과가 적은지:** * **Good:** 단순한 요약, 번역, 상식 수준의 질문 등 모델이 이미 잘 아는 보편적인 작업을 할 때 빠르고 효율적입니다.
  * **Bad:** 출력 형식을 엄격하게 맞춰야 하거나, 사내 도메인 지식처럼 모델이 학습하지 않은 특수한 논리를 요구할 때는 환각(Hallucination)이 발생하기 쉽습니다.

### ② Few-shot Prompting (퓨샷 프롬프팅)
* **한 줄 정의:** 모델이 참고할 수 있도록 질문과 답변의 '예시(Shot)'를 몇 개 제공하여, 원하는 출력 패턴을 학습시키는 기법입니다.
* **핵심 원리:** 프롬프트 입력창(Context Window)에 예시 패턴을 넣어줌으로써, 모델이 다음 토큰을 예측할 때 사용자가 원하는 특정 포맷이나 추론 방식 쪽으로 확률 분포를 강하게 쏠리게(Conditioning) 만듭니다.
* **사용 예시:** > 상태: CrashLoopBackOff -> 조치: 파드 로그 확인
  > 상태: Pending -> 조치: 노드 리소스 및 PVC 바인딩 확인
  > 상태: ImagePullBackOff -> 조치: ?
* **언제 쓰면 좋은지 / 효과가 적은지:**
  * **Good:** JSON 변환, 특정 어투로 답변하기, 로그 분류 등 '패턴'과 '형식'이 중요할 때 매우 효과적입니다.
  * **Bad:** 단순 패턴 매칭을 넘어선 복잡한 수학적 계산이나 다단계 논리 추론이 필요한 문제에서는 예시를 줘도 틀릴 확률이 높습니다.

### ③ Chain-of-Thought, CoT (생각의 사슬)
* **한 줄 정의:** 모델에게 정답을 바로 내놓게 하지 않고, "단계별로 생각해 봐(Think step-by-step)"라고 지시하여 중간 논리 과정을 전개하도록 유도하는 기법입니다.
* **핵심 원리:** 자기회귀(Autoregressive) 모델은 한 번 출력한 단어를 되돌릴 수 없습니다. 바로 정답 토큰을 예측하게 하면 틀리기 쉽지만, 중간에 '추론하는 과정'의 토큰들을 길게 생성하게 만들면, 이 길어진 문맥 자체가 다음 토큰(최종 정답)을 정확하게 예측하는 튼튼한 발판(Context)이 됩니다.
* **사용 예시:** `"특정 네임스페이스의 외부 트래픽을 차단하는 K8s NetworkPolicy를 짤 건데, 바로 코드를 쓰지 말고 어떤 레이블을 선택하고 Ingress/Egress 중 무엇을 제어할지 먼저 논리적으로 단계를 나누어 설명한 뒤에 최종 YAML을 작성해."`
* **언제 쓰면 좋은지 / 효과가 적은지:**
  * **Good:** 복잡한 조건이 얽힌 데이터 분석(예: 의료급여 본인부담률), 수학 문제, 엄격한 보안 정책 설계 등 복합적인 추론이 필요할 때 압도적인 성능을 냅니다.
  * **Bad:** 단순한 정보 검색이나 인사말 같은 쉬운 작업에 쓰면 오히려 응답 시간만 길어지고 토큰 비용이 낭비됩니다.

### ④ Self-Consistency (자기 일관성)
* **한 줄 정의:** 동일한 프롬프트(주로 CoT)를 여러 번 반복 실행하여 다양한 추론 경로를 생성한 뒤, 가장 많이 나온 결론을 최종 정답으로 채택하는 다수결 기법입니다.
* **핵심 원리:** 언어 모델은 확률에 따라 매번 조금씩 다른 텍스트를 생성합니다(Temperature > 0일 때). 한 번의 잘못된 예측(탐욕적 디코딩의 한계)으로 오답에 빠지는 것을 막기 위해, 여러 개의 다양한 추론 경로를 샘플링하고 그중 가장 지배적인 답변을 골라 신뢰도를 극대화합니다.
* **사용 예시:** 복잡한 CKS 보안 취약점 분석을 모델에게 5번 시킵니다. 3번은 '컨테이너 권한 축소'를, 2번은 '네트워크 정책 변경'을 원인으로 꼽았다면, 다수결에 따라 '컨테이너 권한 축소'를 최종 결론으로 도출합니다.
* **언제 쓰면 좋은지 / 효과가 적은지:**
  * **Good:** 절대 틀리면 안 되는 고부가가치 작업이나, 정답이 명확히 존재하는 복잡한 논리 문제에서 정확도를 극한으로 끌어올릴 때 씁니다.
  * **Bad:** API 호출 비용과 대기 시간이 배로 늘어나므로, 실시간성이 중요한 서비스나 가벼운 챗봇에는 부적합합니다.

## 2. 추가 기법 (선택 조사)

**ReAct (Reasoning and Acting)**
* **개요:** 단순한 텍스트 생성을 넘어, 모델이 '생각(Reasoning)'을 하고 그에 맞는 '행동(Action, 외부 도구 사용)'을 번갈아 가며 수행하게 만드는 기법입니다.
* **동작 방식:** 모델이 "현재 K8s 클러스터 상태를 알아야겠군(Thought) -> `kubectl get pods` 명령어를 실행하자(Action) -> (실행 결과 반환) -> 결과를 보니 A 파드가 문제네(Observation)" 형태로 사고하고 행동합니다. LLM을 자율 에이전트(Agent)처럼 활용하는 최신 핵심 방법론입니다.

## 3. 실습 과제 예측 (가설 설정)

*(의료급여 본인부담률 데이터를 바탕으로 세운 논리적 가설입니다. 실습 전 README에 이대로 적으시면 됩니다.)*

1. **Zero-shot(Step 1) 정답률 예측:**
   * **예측:** 약 30~40% 수준으로 다소 낮을 것으로 예측합니다.
   * **이유:** 의료급여 본인부담률은 '연령', '수급권 1종/2종 여부', '질환 종류(틀니 등)' 등 여러 조건이 복합적으로 교차하는 표(Table) 형태의 논리 구조를 띱니다. Zero-shot은 복잡한 조건을 한 번에 계산하는 능력이 떨어지므로 오답률이 높을 것입니다.

2. **가장 큰 개선이 일어날 난이도 예측:**
   * **예측:** **Hard 난이도**에서 가장 극적인 개선이 일어날 것입니다.
   * **이유:** Easy 난이도는 문서에서 단일 조건만 찾으면 되므로 기법 간 차이가 크지 않습니다. 반면 Hard 난이도는 3~4개의 조건을 꼬리물기로 필터링해야 하므로, CoT 기법이 적용되어 다단계 추론을 시작하는 순간 틀리던 문제들을 대거 맞추게 될 것입니다.

3. **가장 큰 정답률 점프가 일어날 구간 예측:**
   * **예측:** **Few-shot → CoT** 로 넘어가는 구간입니다.
   * **이유:** Few-shot은 출력 '형식'을 잡아줄 뿐 복잡한 논리 구조를 해결해주지는 못합니다. 하지만 CoT를 통해 "1) 나이를 확인한다, 2) 수급권 종류를 매칭한다, 3) 예외 조항을 본다"처럼 과정을 쪼개주면(Autoregressive의 문맥 강화), 표 데이터를 해석하는 능력이 근본적으로 향상되어 정답률이 폭발적으로 뛸 것입니다.
