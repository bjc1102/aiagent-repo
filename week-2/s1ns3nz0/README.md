# Part 1: 이론 과제

## 1. 프롬프트 기법 4가지 정리

### 1-1. Zero-shot Prompting

**한 줄 정의:** 예시 없이 지시문만으로 모델에게 작업을 수행시키는 기법.

**핵심 원리:**
대규모 데이터로 사전 학습된 LLM은 instruction tuning과 RLHF를 통해 지시를 따르는 능력을 갖추고 있다. Autoregressive 생성 방식에서 모델은 이전 토큰들(= 지시문)을 조건으로 다음 토큰을 생성하므로, 명확한 지시만으로도 원하는 출력 패턴을 유도할 수 있다.

**사용 예시:**
```
텍스트를 중립, 부정, 긍정으로 분류합니다.
텍스트: 휴가는 괜찮을 것 같아요.
감정:
```
→ 모델이 "중립"이라고 분류. 예시를 제공하지 않았지만 사전 학습된 감정 분석 능력으로 올바르게 답변.

**언제 쓰면 좋은지:**
- 단순 분류, 번역, 요약 등 잘 정의된 작업
- 빠른 프로토타이핑이 필요할 때

**언제 효과가 적은지:**
- 복잡한 다단계 추론이 필요한 경우
- 특수한 출력 형식을 정확히 지켜야 할 때
- 도메인 특화 지식이 필요한 작업

---

### 1-2. Few-shot Prompting

**한 줄 정의:** 프롬프트에 소수의 예시(데모)를 포함시켜 모델이 패턴을 학습하고 유사한 작업을 수행하도록 유도하는 기법.

**핵심 원리:**
Autoregressive 모델에서 프롬프트에 포함된 예시들은 컨텍스트로 작용하여, 모델이 다음 토큰을 생성할 때 "입력 → 출력" 패턴을 조건부로 따르게 만든다. 이를 "in-context learning"이라 하며, 모델의 가중치를 수정하지 않고도 행동을 변경할 수 있다. 흥미롭게도 무작위 레이블을 사용한 예시조차 레이블이 없는 것보다 성능이 좋은데, 이는 예시가 출력 형식과 레이블 공간을 알려주기 때문이다.

**사용 예시:**
```
"awsome" → great  (긍정)
"terrible" → bad  (부정)
"happy" → ?
```
→ 모델이 "great (긍정)"과 유사한 패턴으로 응답. 단어 → 동의어 + 감정 형식을 학습.

**언제 쓰면 좋은지:**
- 특정 출력 형식을 정확히 따라야 할 때
- 도메인 특화 분류/변환 작업
- Zero-shot으로 충분하지 않을 때

**언제 효과가 적은지:**
- 복잡한 수학적·논리적 추론이 필요한 작업 (예: 홀수 합 계산)
- 예시만으로는 추론 과정을 전달하기 어려운 경우

---

### 1-3. Chain-of-Thought (CoT) Prompting

**한 줄 정의:** 모델에게 최종 답변 전에 단계별 추론 과정을 명시적으로 출력하도록 유도하여 복잡한 추론을 가능하게 하는 기법.

**핵심 원리:**
Autoregressive 생성에서 모델은 이전에 생성한 토큰을 기반으로 다음 토큰을 예측한다. 중간 추론 단계를 출력하면, 그 추론 결과가 이후 토큰 생성의 컨텍스트가 되어 더 정확한 최종 답변에 도달할 수 있다. 즉, "생각하면서 쓰기"가 가능해지는 것이다.

- **Zero-shot CoT:** "단계별로 생각해보자(Let's think step by step)" 한 문장만 추가
- **Few-shot CoT:** 추론 과정이 포함된 예시를 함께 제공

**사용 예시:**
```
Q: 집합 {4, 8, 9, 15, 12, 2, 1}에서 홀수를 모두 더하면 짝수인가?
A: 홀수: 9, 15, 1. 합: 9+15+1 = 25. 25는 홀수. 답: 아니오.
```
→ Few-shot으로만 예시를 보여줬을 때 틀리던 문제를, 추론 과정을 포함시키니 정확하게 풀 수 있음.

**언제 쓰면 좋은지:**
- 산술, 상식 추론, 기호 추론 등 다단계 추론 작업
- 조건이 복잡하게 얽힌 판단 (예: 본 과제의 의료급여 본인부담률)

**언제 효과가 적은지:**
- 단순 분류/변환 등 추론이 불필요한 작업 (오히려 비용만 증가)
- 소규모 모델에서는 효과가 제한적

---

### 1-4. Self-Consistency

**한 줄 정의:** 동일 질문에 대해 CoT를 여러 번 수행한 뒤, 가장 빈번한 답변을 다수결 투표로 선택하는 기법.

**핵심 원리:**
Autoregressive 생성은 확률적이므로 temperature를 올리면 매번 다른 추론 경로가 생성된다. 각 경로가 서로 다른 실수를 할 수 있지만, 올바른 답변은 여러 경로에서 일관되게 나타날 가능성이 높다. 즉, "다양한 사고 경로를 탐색하고 가장 일관된 결론을 선택"함으로써 단일 greedy decoding의 한계를 극복한다.

**사용 예시:**
여동생 나이 문제에서 단일 CoT는 35세라고 잘못 답할 수 있지만, 5번 생성하면 67세가 3번, 35세가 2번 나와 다수결로 정답(67세)을 선택할 수 있다.

**언제 쓰면 좋은지:**
- CoT에서 간헐적으로 오답이 나오는 작업
- 정확도가 중요한 고위험 의사결정

**언제 효과가 적은지:**
- API 비용과 지연시간이 중요한 경우 (n번 호출 필요)
- temperature가 너무 높으면 엉뚱한 답, 너무 낮으면 다양성 부족

## 2. 추가 기법

### 2-1. ReAct (Reasoning + Acting)

**정의:** 추론(Thought)과 행동(Action)을 번갈아 수행하며, 외부 도구(검색, API 등)와 상호작용하여 답을 찾는 프레임워크.

**동작 원리:**
1. **Thought** — 현재 상황을 추론하고 다음 행동 계획
2. **Action** — 외부 소스(Wikipedia, 검색 엔진 등)에서 정보 조회
3. **Observation** — 행동 결과를 관찰하고 다시 추론

이 루프를 반복하여 모델이 자체 지식의 한계를 외부 도구로 보완한다. 환각(hallucination)을 줄이고 더 사실에 기반한 답변을 생성할 수 있다.

### 2-2. Prompt Chaining

**정의:** 복잡한 작업을 여러 하위 작업으로 분할하고, 각 프롬프트의 출력을 다음 프롬프트의 입력으로 연결하는 기법.

**동작 원리:**
단일 프롬프트로 처리하기 어려운 복잡한 작업을 파이프라인으로 구성한다. 예를 들어 문서 QA에서:
1. 1단계: 문서에서 관련 인용구 추출
2. 2단계: 추출된 인용구를 바탕으로 최종 답변 생성

각 단계를 독립적으로 디버깅하고 최적화할 수 있어 투명성과 제어가 향상된다.

## 3. 실습 과제 예측 (가설)

### 가설 1: Zero-shot 정답률 예측

**예측: 약 50~60%**

근거:
- easy 문항(5건)은 단순 표 조회로 대부분 맞출 수 있음 → ~80% 이상
- medium 문항(10건)은 다중 조건 판단이 필요하여 일부 실패 → ~50~60%
- hard 문항(15건)은 본인부담률 찾기 + 금액 계산까지 필요하여 실패율이 높음 → ~30~40%
- 특히 참조 데이터를 system prompt에 넣어주므로, 단순 조회는 가능하지만 조건 조합 추론에서 오류 발생 예상

### 가설 2: 가장 큰 개선이 일어날 난이도

**예측: hard 난이도**

근거:
- easy는 이미 높은 정답률 → 개선 여지 적음
- medium은 조건 판단 실수가 주원인 → CoT로 일부 개선
- hard는 다중 조건 판단 + 금액 계산이 결합 → CoT의 단계별 추론이 가장 큰 효과를 발휘할 영역. 계산 과정을 명시적으로 출력하면 계산 오류가 크게 줄 것으로 예상

### 가설 3: 가장 큰 정답률 점프 구간

**예측: Zero-shot → Few-shot 또는 Few-shot → CoT**

근거:
- Zero-shot → Few-shot: 예시를 통해 출력 형식과 조건 판단 패턴을 학습시키면, "어떤 표를 봐야 하는지" 힌트를 줄 수 있어 medium 문항에서 큰 개선 예상
- Few-shot → CoT: hard 문항의 금액 계산에서 단계별 추론이 계산 정확도를 크게 높일 것. 특히 "본인부담률 찾기 → 금액 곱하기"를 명시적으로 분리하면 개선이 클 것으로 예상
- CoT → Self-Consistency: CoT에서 간헐적 오류를 다수결로 보정하므로 소폭 개선 예상 (~5% 이내)

## 각 Step별 프롬프트 전문

### Step 1: Zero-shot

```
아래는 의료급여 본인부담률 참조 데이터입니다.
질문에 대해 정확한 본인부담률을 답하세요. 답만 간결하게 작성하세요.

[참조 데이터 전문 - 01~08번 표]

반드시 아래 JSON 형식으로만 응답하세요:

```

### Step 2: Few-shot

Step 1의 프롬프트에 아래 3개 예시 추가:

```
예시 1)
Q: 1종 수급권자가 입원하면 본인부담률은?
A: {"answer": "무료", "reason": "01번 표 → 1종 → 입원 = 무료"}

예시 2)
Q: 2종 수급권자인 3세 아동이 입원하면 본인부담률은?
A: {"answer": "무료", "reason": "02번 15세 이하 아동 → 입원 → 6세 미만 = 무료"}

예시 3)
Q: 2종 수급권자인 조현병 환자가 외래 진료를 받으면 본인부담률은?
A: {"answer": "병원급 이상 5%", "reason": "07번 정신질환 외래진료 → 조현병 = 병원급 이상 5%"}
```

### Step 3: Chain-of-Thought (CoT)

Step 2의 프롬프트에 아래 단계별 추론 지시 추가:

```
답변 전에 반드시 아래 단계를 따라 추론하세요:
1. 수급권자 종별 (1종/2종) 확인
2. 해당되는 특수 조건 확인 (나이, 질환, 임신 등)
3. 적용할 표 번호 결정 (01~08번)
4. 해당 표에서 정확한 본인부담률 찾기
5. 금액 계산이 필요한 경우 계산 수행

JSON 형식:
{"reasoning_steps": ["단계1", ...], "answer": "답변", "reason": "근거"}
```

### Step 4: Self-Consistency

Step 3의 CoT 프롬프트를 동일하게 사용하되:
- `temperature=0.7`로 설정
- 동일 질문에 대해 **5회 생성**
- **다수결 투표**로 최종 답변 결정

## 오답 분석 — 가장 어려웠던 문항 3건

### 1. q15 (medium) — 모든 Step에서 오답

- **질문:** 65세 이상 2종 수급권자가 임플란트를 할 때 본인부담 보상제나 상한제가 적용되나요?
- **정답:** `해당되지 않음`
- **모델 답변:** `적용되지 않습니다` / `적용되지 않음`
- **원인:** 의미적으로는 정답이지만, answer_key의 `해당되지 않음`과 문자열이 불일치. **표현 불일치 문제**로, 정답 비교 로직의 한계. 실제로는 모델이 정확한 판단을 하고 있음.

### 2. q26, q28 (hard) — 복합 답변 형식 불일치

- **q26 질문:** 틀니(1,500,000원)와 임플란트(1,000,000원) 각각의 본인부담금은?
- **정답:** `틀니 75,000원, 임플란트 100,000원`
- **모델 답변:** `틀니 본인부담금: 75,000원, 임플란트 본인부담금: 100,000원`
- **원인:** 금액 자체는 정확하지만, "본인부담금:" 같은 추가 수식어가 포함되어 문자열 매칭 실패. Few-shot에서는 예시의 간결한 형식을 학습하여 정답 처리됨.

### 3. q09 (medium) — Few-shot에서만 오답

- **질문:** 2종 수급권자인 생후 8개월 아기가 제2차의료급여기관에서 외래 진료를 받으면?
- **정답:** `5%` (1세 미만 → 제2·3차 = 5%)
- **모델 답변 (Few-shot):** `무료`
- **원인:** Few-shot 예시 중 "3세 아동 입원 = 무료" 패턴에 과적합되어, 아동이면 무료라고 잘못 일반화. 제1차 vs 제2차 기관 구분을 놓침.

### 4. q05 (easy) — CoT에서만 오답

- **질문:** 2종 수급권자의 치매 환자가 입원하면 본인부담률은?
- **정답:** `5%` (치매질환 입원 = 5%)
- **모델 답변 (CoT):** `10%`
- **원인:** CoT 추론 과정에서 "2종 수급권자 입원 = 10%" 규칙을 먼저 적용하고, 치매 특례(5%)를 무시함. 단계별 추론이 오히려 일반 규칙에 고착되는 역효과.

## 정답률 개선 과정과 인사이트

### Step 1 → Step 2 (Zero-shot → Few-shot): 86.7% → 93.3% (+6.6%p)

**핵심 개선:** hard 난이도 86.7% → 100% (2건 추가 정답)
- q26, q28의 복합 답변 형식을 예시를 통해 학습하여, 불필요한 수식어 없이 간결하게 답변
- Few-shot의 핵심 효과가 **출력 형식 통일**에 있음을 확인

**부작용:** q09에서 과적합 발생 (Zero-shot에서는 맞았으나 Few-shot에서 틀림)

### Step 2 → Step 3 (Few-shot → CoT): 93.3% → 86.7% (-6.6%p)

**역설적 하락:**
- medium이 80% → 90%로 개선 (q12 형식 문제 해결)
- 그러나 easy에서 q05 오답, hard에서 q26/q28 다시 형식 불일치
- CoT의 장황한 추론이 오히려 **출력 형식을 불안정**하게 만듦
- 이 결과는 "To CoT or not to CoT?" 논문의 지적과 일치: CoT는 수학/기호 추론에 효과적이지만, 단순 조건 검색에서는 오히려 방해가 될 수 있음

### Step 3 → Step 4 (CoT → Self-Consistency): 86.7% → 95.8%* (+9.1%p)

**다수결의 위력:**
- q05에서 5회 중 3회가 정답(5%), 2회가 오답(10%) → 다수결로 정답 선택
- CoT의 간헐적 오류를 효과적으로 보정
- 대부분의 문항에서 5/5 만장일치로 안정적

### 전체 인사이트

1. **Claude Sonnet 4는 기본 성능이 매우 높다**: Zero-shot만으로 86.7%를 달성. TASK.md의 힌트("Zero-shot에서 hard 문항을 거의 못 맞추는 것은 정상")와 달리 hard에서도 86.7% 달성.

2. **Few-shot의 가장 큰 효과는 형식 학습**: 추론 능력보다 출력 형식을 맞추는 데 효과적. 복합 답변(q26, q28)의 형식을 통일시킴.

3. **CoT는 만능이 아니다**: 표 조회 기반의 조건부 판단에서는 오히려 역효과. 단순 조회를 과도한 추론으로 복잡하게 만들 수 있음.

4. **Self-Consistency가 가장 안정적**: 간헐적 오류를 다수결로 보정하는 효과가 뚜렷. 비용은 5배이지만 정확도가 가장 높음.

5. **정답 비교 방식이 중요**: q15처럼 의미적으로 동일한 답변이 형식 차이로 오답 처리되는 경우 다수. 실무에서는 LLM 기반 정답 비교나 시맨틱 매칭이 필요.

## 가설 vs 실제 결과 비교

### 가설 1: Zero-shot 정답률

| | 가설 | 실제 |
|--|------|------|
| 전체 | 50~60% | **86.7%** |
| easy | ~80% | **100%** |
| medium | ~50~60% | **80%** |
| hard | ~30~40% | **86.7%** |

**분석:** Claude Sonnet 4의 표 조회 + 계산 능력을 과소평가했다. 참조 데이터를 구조화된 마크다운 표로 제공한 것이 모델의 조건 판단을 크게 도왔다. 오답 4건 중 3건은 실제로는 추론은 맞았으나 출력 형식 불일치.

### 가설 2: 가장 큰 개선 난이도

| | 가설 | 실제 |
|--|------|------|
| 예측 | hard | **hard** (일부 맞음) |

**분석:** 가설이 부분적으로 맞음. Few-shot에서 hard가 86.7% → 100%로 가장 큰 개선을 보임. 다만 개선 원인은 추론 능력이 아니라 출력 형식 학습이었다.

### 가설 3: 가장 큰 정답률 점프 구간

| | 가설 | 실제 |
|--|------|------|
| 예측 | Zero-shot → Few-shot 또는 Few-shot → CoT | **Zero-shot → Few-shot** (+6.6%p) |

**분석:** Zero-shot → Few-shot이 가장 큰 양(+)의 점프. Few-shot → CoT는 오히려 하락(-6.6%p). CoT → Self-Consistency가 +9.1%p로 실제 가장 큰 점프였으나, 이는 API 에러가 없었다면 달라졌을 수 있다.
