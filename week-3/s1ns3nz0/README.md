## 1. 이론 과제: RAG의 구성 요소

### 1-1. RAG란?

**RAG(Retrieval-Augmented Generation)** 는 LLM의 생성 능력에 외부 문서 검색을 결합하여, 모델이 학습하지 않은 최신·전문 정보도 정확하게 답변할 수 있도록 하는 기법이다.

**왜 필요한가?**
- LLM은 학습 데이터 이후의 정보를 모르고, 전문 도메인 지식이 부족할 수 있다.
- 2주차에서 사용한 "system prompt에 전체 데이터 삽입" 방식은 토큰 한도에 걸리며, 데이터가 커지면 비용과 지연이 급격히 증가한다.

**2주차 방식 vs RAG 방식 비교:**

| 항목 | System Prompt 방식 (2주차) | RAG 방식 |
|------|---------------------------|----------|
| 데이터 규모 | 소규모만 가능 (토큰 한도) | 대규모 문서도 처리 가능 |
| 비용 | 매 호출마다 전체 데이터 전송 → 비용 높음 | 필요한 청크만 전송 → 비용 절감 |
| 정확도 | 데이터가 크면 관련 없는 내용이 노이즈 | 관련 청크만 검색하여 정확도 향상 |
| 확장성 | 제한적 | 문서 추가/업데이트 용이 |

**RAG 파이프라인 전체 흐름:**
```
[Indexing]  문서 → 청킹(Chunking) → 임베딩(Embedding) → 벡터 저장소(Vector Store) 저장
[Retrieval] 질문 → 임베딩 → 벡터 유사도 검색 → 관련 청크 추출 → LLM에 컨텍스트로 전달 → 답변 생성
```

### 1-2. RAG에서 사용되는 용어

| 용어 | 정의 | 상세 |
|------|------|------|
| **Chunking** | 문서를 작은 단위(청크)로 분할하는 과정 | `chunk_size`는 각 청크의 최대 길이, `chunk_overlap`은 인접 청크 간 겹치는 부분으로 문맥 유실을 방지한다. |
| **Embedding** | 텍스트를 고정 차원의 벡터(숫자 배열)로 변환하는 과정 | 대표 모델: OpenAI `text-embedding-3-small`, HuggingFace `multilingual-e5-small`, `bge-m3` 등. 의미적 유사성을 벡터 거리로 측정할 수 있게 한다. |
| **Vector Store** | 벡터를 저장하고 유사도 검색을 수행하는 저장소 | FAISS(Facebook, 로컬/고속), Chroma(간편한 API), Pinecone(클라우드) 등이 있다. |
| **Retriever** | 질문 벡터와 유사한 청크를 검색하는 컴포넌트 | Top-K는 유사도 상위 K개의 청크를 반환하는 것을 의미한다. K가 크면 재현율이 높지만 노이즈가 증가한다. |
| **Generation** | 검색된 청크를 컨텍스트로 활용하여 LLM이 최종 답변을 생성하는 단계 | 프롬프트에 검색 결과를 삽입하여 LLM이 근거 기반 답변을 생성하도록 한다. |

### 1-3. 외부 데이터 소스 기반 응답 생성

**LLM 단독 응답 vs RAG 기반 응답:**
- **LLM 단독**: 학습 데이터에 기반하여 답변. 최신 정보 부재, 도메인 특화 데이터 부족 → 할루시네이션 위험 높음.
- **RAG 기반**: 질문 → 임베딩 → 벡터 검색 → 관련 청크를 컨텍스트에 삽입 → LLM이 근거 기반 답변 생성. 출처를 명시할 수 있어 신뢰성 향상.

**할루시네이션 방지에서 RAG의 역할:**
RAG는 LLM이 "모르는 것을 지어내는" 문제를 완화한다. 검색된 실제 문서를 근거로 답변하므로, 존재하지 않는 정보를 생성할 가능성이 줄어든다.

### 1-4. LangChain 기반 RAG 파이프라인 구조

| 단계 | LangChain 컴포넌트 | 본 과제에서 사용한 것 |
|------|-------------------|---------------------|
| 문서 로딩 | Document Loader | `PyPDFLoader` |
| 청킹 | Text Splitter | `RecursiveCharacterTextSplitter` (chunk_size=500, overlap=100) |
| 임베딩 | Embeddings | `HuggingFaceEmbeddings` (`multilingual-e5-small`) |
| 벡터 저장 | Vector Store | `FAISS` |
| 검색 + 생성 | Retriever + Chain | `similarity_search` (Top-K=3) |

## 2. 실습 과제 결과

### 2-1. Golden Dataset 구축 과정

PDF를 직접 읽고, 의료급여 본인부담률 표를 중심으로 5개 질문을 설계했다.

**질문 설계 시 고려한 점:**
- **easy**: 표에서 단일 조건으로 바로 찾을 수 있는 값 (1종 외래 의원 본인부담금, 2종 틀니 부담률)
- **medium**: 2~3개 조건 조합 (종별 + 진료유형 + 기관유형, 나이 + 진료유형)
- **hard**: 다중 조건이 교차하는 추나요법 표에서 질환유형 + 추나유형 + 종별을 모두 조합

### 2-4. 검색 실패 원인 분석

**q03 (medium, 2종 입원 종합병원 10%) - 실패:**
- 본인부담률 표가 PDF에서 구조화된 표 형태이나, PyPDFLoader가 이를 plain text로 추출할 때 행/열 관계가 깨짐
- "2종", "입원", "10%" 등의 키워드가 하나의 청크에 함께 포함되지 않아 유사도 검색에서 누락

**q04 (medium, 6세 미만 아동 입원 무료) - 실패:**
- 실제로 관련 청크가 Top-1에 검색되었으나("6세 미만의 아동은 식대를 포함한 입원 진료비가 면제 적용됩니다"), 키워드 매칭 로직에서 "입원 6세 미만 무료"라는 연속 문자열로 비교했기 때문에 실패로 판정됨
- **실질적으로는 검색 성공에 가까움** — 평가 로직의 한계

**q05 (hard, 추나요법 복합 조건) - 실패:**
- 추나요법 본인부담률 표는 PDF에서 복잡한 행/열 구조를 가지며, PyPDFLoader 추출 시 표 구조가 완전히 소실됨
- "디스크", "복잡추나", "40%" 등의 키워드가 청크에 존재하지 않아 의미적 유사도가 낮음
- 표 형태 데이터에 대해 `pdfplumber` 같은 표 전용 추출기 사용이 필요

## 3. 실습 전 가설 vs 실제 결과

### 가설 1: 의료급여 PDF 청킹 시 가장 큰 어려움

**가설:** 의료급여 PDF는 표(table) 형태 데이터가 많아, 고정 크기 청킹 시 표의 행/열 관계가 깨져서 정보 손실이 발생할 것이다.

**실제 결과:** ✅ **가설 일치.** PyPDFLoader로 추출 시 표의 셀 경계가 사라지고, 행과 열이 뒤섞인 plain text가 되었다. 특히 본인부담률 표에서 "1종/2종", "입원/외래", "1차/2차/3차" 축의 교차 관계가 소실되어, 검색 시 정확한 조건 조합을 찾기 어려웠다.

### 가설 2: 검색 실패가 많을 난이도

**가설:** hard 난이도 질문에서 검색 실패가 가장 많을 것이다. 다중 조건이 교차하는 질문은 관련 정보가 여러 표/섹션에 분산되어 있어, 단일 청크에서 모든 근거를 찾기 어렵기 때문이다.

**실제 결과:** ✅ **가설 부분 일치.** hard(q05)는 예상대로 완전 실패했다. 다만 medium(q03, q04)도 실패하여, 표 구조 소실 문제는 난이도와 무관하게 발생했다. easy 질문이 성공한 이유는 관련 키워드가 FAQ 텍스트 등 비표 영역에도 존재했기 때문이다.

## 4. 인사이트 및 다음 단계 준비

### 핵심 인사이트

1. **표 데이터의 청킹 전략이 핵심이다.** 고정 크기 텍스트 분할은 표 형태 데이터에 치명적이다. `pdfplumber`로 표를 구조화된 텍스트(마크다운 등)로 변환한 뒤 청킹하는 것이 효과적일 것이다.
2. **임베딩 모델의 한국어 성능이 중요하다.** `multilingual-e5-small`은 다국어를 지원하지만 한국어 전문 용어(수급권자, 추나요법 등)에 대한 임베딩 품질이 제한적일 수 있다. 한국어 특화 모델(`ko-sroberta-multitask` 등) 비교가 필요하다.
3. **평가 로직도 개선이 필요하다.** 문자열 매칭 기반 평가는 의미적으로 성공인 경우도 실패로 판정할 수 있다. LLM 기반 평가(정답과 검색 결과의 의미적 일치 판정)가 더 정확할 것이다.

### Retrieval + Generation 단계 준비 사항

- 표 데이터 전용 로더(pdfplumber) 적용으로 청킹 품질 개선
- Top-K 값 조정 (K=5 이상) 및 reranking 적용 검토
- 검색된 청크 + 질문을 프롬프트에 결합하여 LLM(Claude 등)으로 최종 답변 생성하는 파이프라인 구축
- 답변의 출처(source_section)를 함께 제공하여 검증 가능한 답변 생성
