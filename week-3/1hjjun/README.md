## 1. RAG란?

- **정의:** 모델의 파라미터 내에 없는 외부 데이터를 검색(Retrieval)하여, 그 내용을 바탕으로 답변을 생성(Generation)하는 기술
- **필요성:** 모델 학습 이후의 최신 정보 반영, 특정 도메인(의료, 사내 문서 등) 지식 활용, 그리고 근거 기반 답변을 통한 할루시네이션 방지를 위해 필요

### **2주차(Full Context) 방식과의 차이:**

- **2주차:** 시스템 프롬프트에 모든 데이터를 넣는 방식. 데이터 양이 많아지면 비용이 급증하고 모델의 문맥 제한(Context Window)을 초과함.
- **RAG:** 수만 페이지 중 질문과 관련된 '핵심 조각'만 골라 넣기 때문에 훨씬 효율적이고 대규모 데이터 처리가 가능함.

---

## 2. RAG 주요 용어 정리

| 용어             | 조사 내용                                                                                                                                |
| :--------------- | :--------------------------------------------------------------------------------------------------------------------------------------- |
| **Chunking**     | 긴 문서를 의미 있는 단위로 자르는 과정. `chunk_size`는 조각의 크기, `overlap`은 문맥 유실 방지를 위해 조각 간 겹치는 부분의 길이를 뜻함. |
| **Embedding**    | 텍스트의 의미를 고차원 숫자의 집합(벡터)으로 변환하는 과정. OpenAI, HuggingFace 등의 모델이 지도를 그려주는 역할을 함.                   |
| **Vector Store** | 변환된 벡터와 원본 텍스트를 저장하고, 질문과 가장 유사한 좌표를 빠르게 찾아주는 특수 DB (FAISS, Chroma 등).                              |
| **Retriever**    | 저장소에서 질문과 관련된 상위 K개(`Top-K`)의 청크를 뽑아오는 컴포넌트. K가 클수록 정보량은 많아지지만 노이즈도 섞일 수 있음.             |
| **Generation**   | 검색된 청크(Context)를 질문과 함께 LLM에 전달하여 최종 답변을 생성하는 단계.                                                             |

---

## 3. 외부 데이터 기반 응답 생성

- **차이점:** LLM 단독 응답은 모델의 '기억'에만 의존하지만, RAG는 '오픈북 테스트'처럼 외부 참고서를 보고 답한다.
- **할루시네이션 방지:** "모르는 내용은 답변하지 말고 제공된 컨텍스트 내에서만 답하라"는 지침을 줄 수 있어, 근거 없는 말을 지어내는 현상을 획기적으로 줄입니다.

---

## 4. 실습 과제 예측 (가설 설정)

### 가설 1: 의료급여 PDF 청킹의 어려움

- 표 처리: 의료급여 PDF 내의 병합된 셀이 포함된 표는 단순 텍스트 추출 시 데이터 오염이 발생할 것이다. 이를 방지하기 위해 병합된 구조를 해제하고 마크다운(Markdown) 표 형식으로 변환하여 저장함으로써, 행-열 간의 의미 관계를 보존하는 것이 검색 정확도 향상에 필수적일 것이다.
- 소주제 분리 가설: "5개의 소주제가 있는 것으로 확인되므로, 먼저 소주제를 기준으로 전체 글을 5개로 먼저 나눈 다음. 그 후 하나의 주제 안에서 청크 간의 **Overlap(중첩)**을 준다면 주제끼리의 노이즈와 정보의 누락을 동시에 예방할 수 있을 것이다.

### 가설 2: 검색 실패가 많을 질문 유형

**"복합 조건형 질문"**에서 실패가 많을 것으로 예측됩니다.

- 예: "소득인정액이 A원 이하이면서 B질환을 앓고 있는 65세 이상의 혜택은?"
- **이유:** 임베딩 모델은 문장의 전체적인 의미는 잘 잡지만, 숫자(A원)나 구체적인 나이(65세) 같은 **키워드 수준의 정교한 필터링**에는 약하기 때문입니다. 단순 의미 유사도만으로는 조건 중 하나를 놓친 청크를 가져올 확률이 높다.

# 3주차 실습 과제: 의료급여 RAG Indexing 파이프라인 구축 리포트

## 1. 프로젝트 개요

- **프레임워크**: LangChain (LCEL 기반 초경량 구조)
- **임베딩 모델**: `models/gemini-embedding-001`
- **벡터 저장소**: FAISS (로컬 저장 방식)
- **실행 환경**: macOS (MacBook Air), Python 3.13 가상환경(`.venv`)

## 2. Golden Dataset 구축 (Step 1)

질문 설계 시 **단순 수치 조회**부터 **다중 조건 예외 규정**까지 단계별 난이도를 고려하여 7문항을 구축했습니다.

- **고려한 점**:
  - 1종/2종 수급권자 구분에 따른 본인부담률 차이 반영.
  - 6세 미만, 15세 이하 등 연령에 따른 특례 규정 포함.
  - 입원비와 식대가 서로 다른 비율로 적용되는 복합 상황 설정.

> **[파일 확인]** `golden_dataset.jsonl` (총 7문항 포함)

## 3. RAG Indexing 파이프라인 구성 (Step 2)

- **PDF 로딩**: `pdfplumber`를 사용하여 페이지 좌표 기반 추출 수행.
  - **전략**: 2단 구성 PDF의 텍스트 혼선을 막기 위해 페이지를 좌/우(`width/2`)로 분할하여 개별 로딩.
- **청킹 설정**:
  - **방식**: 페이지 및 영역별 단위 청킹. [cite: 1]
  - **특이사항**: 각 청크의 상단에 `[주제: 2024 의료급여제도 안내 및 Q&A]` 메타데이터를 강제 주입하여 검색 맥락 유지.
- **임베딩 및 저장**: `gemini-embedding-001`을 통해 벡터화 후 FAISS 인덱스로 로컬 저장.

## 5. 실습 과제 예측 및 결과 비교 (README 필수 포함)

### [실습 전 가설]

1. **청킹의 어려움**: 2단 구성 PDF에서 왼쪽/오른쪽 글자가 섞여 문맥이 깨질 것으로 예상함.
2. **검색 실패 예측**: 다중 조건이 포함된 `hard` 난이도 질문(`q16`, `q17`)에서 관련 정보가 여러 페이지에 흩어져 있을 경우 검색이 어려울 것으로 예상함.

### [실습 후 결과]

1. **가설 1 검증**: 실제로 일반 로더 사용 시 텍스트가 섞였으나, `pdfplumber`의 `within_bbox` 기능을 사용하여 영역을 강제 분할함으로써 해결함.
2. **가설 2 검증**: 정보가 페이지 경계에 걸쳐 있는 경우를 대비해 `Top-K=3`으로 설정하고 주제 맥락을 본문에 주입한 결과, 복합 조건 질문에서도 필요한 정보를 모두 성공적으로 가져옴을 확인했음.
