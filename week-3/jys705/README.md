# 3주차 과제 (정연승): RAG Indexing 파이프라인 구축 및 이론 과제

## Part 1: 이론 과제 답변

### 1. RAG란?
* **정의 및 필요성**: RAG(Retrieval-Augmented Generation)는 거대 언어 모델(LLM)이 답변을 생성하기 전, 신뢰할 수 있는 외부 지식 베이스에서 관련 정보를 검색(Retrieval)하여 답변에 활용하는 기술입니다. LLM의 학습 데이터가 최신이 아니거나 특정 도메인 지식이 부족할 때 발생하는 할루시네이션(환각)을 방지하기 위해 필수적입니다.
* **2주차 방식과의 차이**: 2주차에는 시스템 프롬프트에 PDF의 표 데이터를 전체 하드코딩하여 주입했습니다. 이 방식은 데이터가 커질수록 컨텍스트 윈도우 한계를 초과하고 토큰 비용이 급증합니다. RAG는 전체 데이터 중 **질문과 관련된 특정 조각(Chunk)만 유동적으로 검색**하여 주입하므로 대규모 문서 처리에 적합합니다.
* **전체 흐름**: 
  * `Indexing`: 문서 로드 → 텍스트 분할(Chunking) → 의미를 숫자로 변환(Embedding) → 벡터 저장소(Vector Store)에 저장
  * `Retrieval`: 사용자 질문 → 질문 임베딩 → 가장 유사한 청크 검색 → LLM에 문맥으로 제공하여 답변 생성

### 2. RAG에서 사용되는 용어
| 용어 | 조사 내용 |
|------|----------|
| **Chunking** | 방대한 문서를 LLM이 처리하기 좋은 크기로 분할하는 과정. `chunk_size`는 조각의 크기를, `chunk_overlap`은 조각이 나뉠 때 문맥이 끊기지 않도록 앞뒤 문장을 겹치게 하는 크기를 의미합니다. |
| **Embedding** | 텍스트의 의미를 컴퓨터가 이해하고 계산할 수 있도록 고차원 숫자 벡터로 변환하는 과정입니다. (예: OpenAI `text-embedding-3-small`) |
| **Vector Store** | 변환된 벡터 데이터를 저장하고, 사용자의 질문 벡터와 코사인 유사도 등을 비교해 가장 의미가 가까운 문서를 빠르게 찾아주는 데이터베이스입니다. (예: FAISS, Chroma) |
| **Retriever** | 질문을 받아 Vector Store에서 가장 유사도가 높은 청크를 검색해오는 컴포넌트입니다. `Top-K`는 검색 결과로 가져올 상위 N개의 문서 개수를 의미합니다. |
| **Generation** | 검색된 청크(문맥)들을 바탕으로 LLM이 사용자에게 최종적인 자연어 답변을 생성하는 단계입니다. |

### 3. 외부 데이터 소스 기반 응답 생성
* **단독 응답 vs 검색 기반 응답**: LLM 단독 응답은 자신이 학습한 과거의 파라미터 메모리에만 의존하므로 부정확할 수 있습니다. 검색 기반 응답은 외부에서 '정답지'를 찾아 쥐여주고 답변하게 하므로 정확도와 출처 추적이 가능합니다.
* **API 호출 흐름**: 질문 입력 → 질문 임베딩 API 호출 → 벡터 DB 유사도 검색 → 검색된 텍스트로 컨텍스트 구성 → 컨텍스트+질문으로 LLM API 호출 → 답변 반환
* **할루시네이션 방지 역할**: LLM에게 "주어진 문서 내용 안에서만 대답해"라는 제약을 걸 수 있어, 모르는 정보를 지어내는 현상을 획기적으로 줄일 수 있습니다.

### 4. LangChain 기반 RAG 파이프라인 구조
| 단계 | LangChain 컴포넌트 |
|------|-------------------|
| 문서 로딩 | `Document Loader` (`PyPDFLoader`, `PDFPlumberLoader` 등) |
| 청킹 | `Text Splitter` (`RecursiveCharacterTextSplitter` 등) |
| 임베딩 | `Embeddings` (`OpenAIEmbeddings` 등) |
| 벡터 저장 | `Vector Store` (`FAISS`, `Chroma` 등) |
| 검색 + 생성 | `Retriever` + `Chain` (또는 `create_retrieval_chain`) |

### 5. 실습 전 가설 설정
1. **청킹의 가장 큰 어려움**: 의료급여 PDF는 복잡한 다단 표(Table)가 많아, 일반적인 텍스트 분할기로 자르면 표의 행렬(Row/Column) 관계가 깨져 문맥이 손실될 가능성이 높습니다.
2. **검색 실패 예측**: **Hard 난이도** (예: 나이 + 수급권자 + 질환 + 복합 조건)에서 실패가 많을 것으로 예상됩니다. 질문의 키워드가 여러 청크에 분산되어 있거나, 표가 분할되었을 때 임베딩 모델이 정확한 수치가 있는 행을 최우선으로 찾지 못할 수 있기 때문입니다.

## Part 2: 실습 과제 결과 (RAG Indexing & 검증)

### 1. 실습 환경 및 사용 기술
* **프레임워크**: LangChain
* **임베딩 모델**: OpenAI `text-embedding-3-small`
* **벡터 저장소**: Chroma DB (로컬 실행)
* **실행 환경**: Python

### 2. Golden Dataset 구축 과정
* **질문 설계 시 고려한 점**: 단순 조건(Easy)부터, 나이/종별/질환/의료기관 등 여러 조건이 복합적으로 얽힌 질문(Hard)까지 다양하게 구성하여 변별력을 높였습니다.
* **증거 텍스트(evidence_text) 추출**: RAG 파이프라인의 기계적 평가를 위해 임의로 문장을 가공하지 않고, PDF 원문에 존재하는 표 안의 텍스트(예: "디스크, 협착증 30%")를 그대로 발췌하여 매칭의 정확도를 높였습니다.

### 3. RAG Indexing 파이프라인 구성
초기에는 `PyPDFLoader`와 큰 청크 크기(1000)를 사용했으나 검색률이 q01 하나만 성공한 20%의 성공률이 나오는 문제가 발생했습니다. 데이터 정제를 위해 아래와 같이 파이프라인 설정을 최적화했습니다.

* **로더 (Loader)**: `PDFPlumberLoader` (과제 힌트 참고. 시각적 표 레이아웃 보존에 강점)
* **청킹 (Text Splitter) 설정**: `RecursiveCharacterTextSplitter`
  * `chunk_size`: 500 (주제 밀집도를 높이기 위해 대폭 축소)
  * `chunk_overlap`: 100 (표나 문장의 단절을 막기 위한 여유분)
* **총 청크 수**: 62개

### 4. 검색 품질 확인 결과 (Top-K=3)

| 질문 ID | 난이도 | 검색 결과 | 검색된 청크 요약 및 분석 |
| :--- | :--- | :--- | :--- |
| **q01** | Easy | ✅ 성공 | 65세 이상 틀니 본인부담률 표가 정확히 최상단에 검색됨. |
| **q02** | Easy | ✅ 성공 | 치매 환자 입원 5% 규정이 누락되지 않고 완벽하게 검색됨. |
| **q03** | Medium | ❌ 실패 | 1세 미만 외래 진료를 물었으나, 다단 표(rowspan) 구조가 텍스트화되며 관계가 모호해져 오답 청크(FAQ의 전원 절차 등)가 상위 노출됨. |
| **q04** | Medium | ❌ 실패<br>(위양성) | **(실제로는 정답 검색이 성공함)** 청크에 "16세 이상 18세 이하 아동인 경우에는 치아홈메우기... 5%를 적용"이라는 완벽한 문장(FAQ 파트)이 도출됨. 그러나 `evidence_text`("16세 이상~18세 이하 5%")와 텍스트 일치 검사에서 조사/기호 차이로 기계적 실패 판정됨. |
| **q05** | Hard | ✅ 성공 | (초기 테스트 0% 원인 문항) 가장 복잡했던 추나요법 데이터가 `PDFPlumber` 및 세밀한 청킹 적용 후 정확히 검색됨. |

* **최종 검색 성공률**: 3/5 (60%) / *※ q04의 위양성을 포함한 논리적 성공률은 80%*

#### 실패 원인 분석
1. **표 데이터 구조 붕괴 (q03)**: 여전히 복잡하게 병합된 다단 표(Nested Table)는 텍스트 파싱 시 헤더와 데이터의 관계가 깨져 벡터 공간에서 의미를 상실하는 한계가 존재합니다.
2. **Exact Match 기반 평가의 한계 (q04)**: 문서 내에 완벽한 정답이 있음에도 파이썬의 단순 문자열 일치 검사로는 이를 실패로 판정했습니다. 실무에서는 RAG 시스템 평가 시 LLM을 고려해놔야 함을 깨달았습니다.

### 5. 인사이트 및 다음 단계 준비 사항
* **가설 검증**: 실습 전 가설(표 청킹의 어려움, Hard 난이도의 실패 위험)은 매우 정확했습니다. 그러나 로더 교체(`pdfplumber`)와 청킹 사이즈 축소를 통해 Hard 문항(q05)의 검색을 성공시키는 유의미한 데이터 파이프라인 최적화를 이뤄냈습니다.
* **인사이트**: RAG의 성능은 LLM 프롬프트보다 데이터를 어떻게 자르고 정제하느냐(Preprocessing)에 달려있음을 2주차에 이어 한번 더 느꼈습니다.
* **다음 단계(Generation) 준비**: 
  1. 검색된 청크 리스트를 Context로 삼아 사용자 질문과 함께 LLM에 주입하는 **Prompt Template** 작성이 필요합니다.
  2. q03번처럼 검색기가 정확한 정보를 가져오지 못했을 때, LLM이 거짓말(Hallucination)을 하지 않고 *"문서에서 근거를 찾을 수 없습니다"*라고 답변할 수 있도록 제어하는 안전장치 마련이 핵심 과제가 될 것입니다.
