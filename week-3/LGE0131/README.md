# 1.이론 과제

## 1.1 RAG란?
RAG(Retrieval-Augmented Generation)는 거대 언어 모델(LLM)의 텍스트 생성 능력에 외부 정보 검색 능력을 결합한 아키텍처입니다. LLM은 사전 학습된 데이터에만 의존하기 때문에 최신 정보를 알지 못하고, 특정 도메인의 전문 지식이 부족하다는 한계가 있습니다. RAG는 신뢰할 수 있는 외부 데이터 소스에서 관련 정보를 실시간으로 검색하여 LLM에 제공함으로써, 이러한 한계를 극복하고 정확도 높은 답변을 생성하기 위해 필요합니다.

### 전체 데이터 삽입 방식(2주차)과 RAG 방식의 차이
시스템 프롬프트에 문서 전체를 삽입하는 방식은 모델의 컨텍스트 윈도우(Context Window) 한계로 인해 대용량 문서를 처리할 수 없으며, API 토큰 사용량이 급증하여 비용이 크게 발생합니다.
반면, RAG 방식은 전체 문서 중 사용자 질문과 의미적으로 가장 유사한 핵심 정보(청크)만 선별하여 프롬프트에 삽입합니다. 이를 통해 토큰 한계를 우회하고 연산 및 비용 효율성을 크게 높일 수 있습니다.

### RAG 파이프라인의 흐름
**1. Indexing (데이터 준비 단계)**
원본 문서 수집 → 일정한 크기로 분할(Chunking) → 텍스트를 임베딩 벡터로 변환 → 벡터 저장소(Vector Store)에 저장

**2. Retrieval (검색 및 생성 단계)**
사용자 질문 입력 → 질문을 임베딩 벡터로 변환 → 벡터 저장소에서 유사도 검색 → 검색된 결과를 프롬프트에 컨텍스트로 추가 → LLM이 최종 답변 생성

## 1.2 RAG에서 사용되는 용어
- **Chunking (청킹):** 문서를 LLM이 처리하기 적합한 크기의 작은 텍스트 조각(청크)으로 분할하는 작업입니다.
    - chunk_size: 하나의 청크에 포함될 최대 글자 수 또는 토큰 수를 지정합니다.
    - chunk_overlap: 분할 시 문맥이 끊기는 것을 방지하기 위해 앞뒤 청크 간에 중복해서 포함할 글자 수나 토큰 수를 지정합니다.

- **Embedding (임베딩):** 자연어 텍스트의 의미를 기계가 계산할 수 있도록 고차원 실수 벡터로 변환하는 기술입니다.
    - 주요 임베딩 모델 비교: OpenAI의 text-embedding-3-small/large 모델은 범용적인 성능과 다국어 처리 능력이 우수하며 API 형태로 쉽게 사용 가능합니다.
    - 반면 BGE, E5 등은 오픈소스 모델로 로컬 환경 구축 및 특정 도메인 파인튜닝에 적합합니다.

- **Vector Store (벡터 저장소):** 임베딩된 벡터 데이터와 메타데이터를 저장하고, 질문 벡터와의 수학적 거리(유사도)를 빠르게 계산하여 검색하는 데이터베이스입니다.
    - FAISS는 메모리 기반으로 로컬에서 빠른 검색을 수행할 때 유리하며,
    - Chroma는 직관적인 API를 제공하는 경량 벡터 데이터베이스로 프로토타이핑에 주로 사용됩니다.

- **Retriever (검색기):** 사용자 질문(쿼리)을 바탕으로 벡터 저장소 등에서 가장 적합한 텍스트 청크를 찾아 반환하는 컴포넌트입니다.
    - Top-K: 유사도 검색 결과 중 가장 관련성이 높은 상위 K개의 청크(문서)를 추출한다는 의미입니다.

- **Generation (생성):** 검색기(Retriever)가 찾은 외부 텍스트 청크를 사용자 질문과 함께 프롬프트로 구성하여 LLM에 전달하고, 최종 답변을 도출하는 단계입니다.

## 1.3 외부 데이터 소스 기반 응답 생성

### LLM 단독 응답 vs 외부 데이터 검색 기반 응답의 차이
LLM 단독 응답은 모델이 학습 과정에서 내부 파라미터에 저장한 지식에만 의존하여 답변을 생성합니다.
반면 외부 데이터 검색 기반 응답은 실시간으로 외부 데이터베이스에서 사실 관계를 확인하고 문서를 가져와 이를 근거로 답변을 생성하므로 정보의 정확성과 출처 추적이 가능합니다.

### API 호출 흐름
1) 사용자 질문 텍스트 수신
2) 임베딩 모델 API를 호출하여 사용자 질문을 고차원 벡터로 변환
3) 벡터 저장소에 질의하여 코사인 유사도 등이 높은 텍스트 청크 검색
4) 검색된 텍스트 청크들을 결합하여 LLM에 전달할 시스템 프롬프트(컨텍스트) 구성
5) 사용자 질문과 구성된 컨텍스트를 포함하여 LLM 생성 API 호출
6) 최종 생성된 텍스트 응답 반환

### RAG가 어떻게 할루시네이션을 방지하는지
할루시네이션(Hallucination)은 LLM이 사실이 아닌 정보를 그럴듯하게 지어내는 현상입니다. RAG는 LLM에 "이 외부 문서(컨텍스트)에 기반해서만 답변을 작성하라"는 제약을 부여합니다. 모델이 내부 지식에 의존해 추측하는 대신 명확히 검색된 정보만을 요약 및 재구성하도록 강제하기 때문에 할루시네이션을 억제하고 신뢰성 있는 응답을 보장합니다.

## 1.4 LangChain 기반 RAG 파이프라인 구조
LangChain 프레임워크를 활용하면 RAG 파이프라인의 각 구성 요소를 모듈화하여 쉽게 구축할 수 있습니다.

- **문서 로딩 (Document Loader):**
    - 외부 소스(PDF, 웹페이지, 텍스트 파일 등)에서 데이터를 읽어와 LangChain의 기본 단위인 Document 객체로 변환합니다.
    - 대표적으로 PyPDFLoader, WebBaseLoader 등이 사용됩니다.

- **청킹 (Text Splitter):**
    - 로드된 긴 Document를 지정된 크기의 청크로 분할합니다.
    - 대표적으로 RecursiveCharacterTextSplitter가 사용되며, 이는 단락, 문장, 단어 순으로 구조를 유지하며 분할하는 데 유리합니다.

- **임베딩 (Embeddings):**
    - 분할된 텍스트 청크를 벡터로 변환하는 인터페이스입니다.
    - OpenAIEmbeddings 클래스 등을 사용하여 OpenAI의 임베딩 API와 연결합니다.

- **벡터 저장 (Vector Store):**
    - 임베딩된 벡터와 원본 텍스트(메타데이터 포함)를 저장하고 관리합니다.
    - LangChain의 FAISS나 Chroma 래퍼(Wrapper) 클래스를 사용하여 from_documents 메서드 등으로 데이터를 저장합니다.

- **검색 + 생성 (Retriever + Chain):**
    - 벡터 저장소를 검색기 인터페이스로 활용하기 위해 .as_retriever() 메서드를 사용합니다.
    - 검색된 문서와 질문을 LLM에 전달하여 답변을 생성하는 체인은 create_retrieval_chain 및 create_stuff_documents_chain을 결합하여 구축합니다.

#### 가설과 실제 결과 비교
PyPDFLoader와 RecursiveCharacterTextSplitter(크기 1000, 오버랩 200)의 조합을 사용하면, 단일 조건 매칭(Easy) 및 다중 조건 매칭(Medium) 문제에 대해서는 높은 정확도로 답변을 생성할 수 있을 것이라 예상했습니다. 반면, 정보가 여러 페이지에 분산되어 있거나 별도의 수치 계산이 필요한 문제(Hard)의 경우 검색된 청크의 결합 실패나 계산 오류로 인해 정답률이 낮을 것으로 가설을 세웠습니다.

예상과 달리 다중 조건을 매칭해야 하는 Medium 난이도(q01, q06, q07)와 일부 Easy 난이도(q04)에서 다수의 실패가 발생했습니다. 오히려 복잡한 계산이나 예외 규정을 묻는 Hard 난이도(q09, q10)에서는 프롬프트 및 넓은 오버랩 영역 덕분에 성공적으로 정답을 도출했습니다.

데이터 파싱에서 한계가 있었던 것 같습니다. 문서 내 다수의 정보가 '표(Table)' 형태로 존재하는데, PyPDFLoader가 표를 단순 텍스트로 평탄화(Flatten)하는 과정에서 행과 열의 의미적 연결이 끊어졌습니다. 그 결과, LLM이 '조건'과 '비율'을 엉뚱하게 매칭하는 현상이 발생하여 Medium 난이도에서의 연쇄적인 오답으로 이어졌습니다.


## 2.3 실행 환경 및 기술 스택
- 프레임워크: LangChain
- 임베딩 모델: OpenAI text-embedding-3-small
- 벡터 저장소: Chroma (Local)
- LLM: OpenAI gpt 5.4 nano
- 실행 환경: Windows 10/11, VS Code, Python 가상환경(myenv)

## 2.4 Golden Dataset 구축 과정

1. 데이터 스키마 일관성 확보
- 자동화된 평가(LLM-as-a-Judge 등) 환경에서 파싱 오류가 발생하지 않도록 데이터 타입을 엄격하게 통일했습니다.
- evidence_text와 conditions 항목은 배열(Array) 형태로, 여러 출처가 합쳐진 source_section 항목은 단일 문자열(String) 형태로 규격화했습니다.

2. 가이드라인에 기반한 난이도 세분화
- 질문 설계 가이드라인의 난이도 기준을 준수하여 문항을 분류했습니다.
- 단일 조건 조회를 요구하는 문항은 'Easy', 2~3개의 조건이 조합된 문항은 'Medium'으로 설정했습니다.
- 다중 조건 확인 후 연산(금액 계산)이 필요하거나, 예외 규정을 파악해야 하는 복합 문항은 'Hard'로 배정했습니다.

3. 검색 성능(Retrieval) 평가에 최적화된 근거 텍스트 작성
- RAG 시스템이 문서 청크(Chunk)에서 정답을 제대로 검색했는지 정확히 평가하기 위해 evidence_text를 작성했습니다.
- 평가 파이프라인의 텍스트 대조(Exact Match 등) 효율을 높이기 위해, 내용을 임의로 요약하지 않고 원문에 존재하는 문장 형태를 최대한 유지했습니다.

4. 엣지 케이스(Edge Case) 및 예외 조항 반영
- 시스템이 일반 규정과 예외 규정을 명확히 구분하여 답변할 수 있는지 검증하기 위한 문항을 포함했습니다.
- 예: 2종 수급권자의 일반 외래 본인부담률(15%)과 특정 질환(조현병 등)의 예외 본인부담률(5%)을 구분하는 문항을 설계하여 검색 및 생성의 정밀도를 테스트했습니다.

## 2.5 RAG Indexing 파이프라인 구성
- 문서 로드 (Loader): PyPDFLoader를 사용했습니다. 원본 문서가 이미지 스캔 기반이며 표(Table) 데이터 비중이 높다는 특징이 있습니다.
- 청킹 설정 (Splitter): RecursiveCharacterTextSplitter를 적용했습니다.
    - chunk_size=1000
    - chunk_overlap=200
    - 표 데이터와 관련된 문맥이 잘려 의미가 훼손되는 것을 방지하기 위해 오버랩 크기를 비교적 여유롭게 설정했습니다...........
- 임베딩 (Embedding): 비용 효율성과 속도를 고려하여 text-embedding-3-small 모델을 사용했습니다.
- 벡터 저장소 (Vector Store): 가벼운 로컬 테스트에 적합한 Chroma를 활용하여 임베딩 데이터를 저장하고 인덱싱했습니다.

## 2.7 인사이트 및 다음 단계 준비 사항
-dml료급여제도와 같이 수치와 표, 예외 조항이 많은 공공문서는 단순 텍스트 기반 RAG 파이프라인만으로는 높은 정확도를 담보하기 어렵습니다.
- 검색(Retrieval) 성능 개선 외에도, 추출된 정보를 바탕으로 추론하는 생성(Generation) 단계의 정교한 통제가 필요할 것 같습니다.

#### 다음 단계 준비 사항
- 문서 처리 고도화: 표(Table) 데이터와 이미지 내 텍스트(OCR)를 구조적으로 읽어올 수 있도록 RapidOCR 또는 Unstructured 기반의 파서를 도입하여 로더를 교체합니다.- 
- 의미 기반 청킹: 단순 글자 수 기반이 아닌, 문서의 논리적 구조나 문단 단위로 분할하는 방식(SemanticChunker 등)을 테스트합니다.
- 생성 프롬프트 엔지니어링 (Generation): 수치 계산 문제 해결을 위해 CoT(Chain of Thought) 프롬프트 기법을 적용하여 모델이 단계별로 연산하도록 유도합니다.
- 평가 체계 정량화: RAGAS 프레임워크를 도입하여 Context Precision(검색 정확도)과 Answer Relevance(답변 관련성)를 정량적 수치로 평가할 준비를 합니다.
