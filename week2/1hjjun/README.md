# 2주차 과제: 프롬프트 엔지니어링으로 LLM 개선하기

## 1. 이론 과제 : 프롬프트 엔지니어링 기법 조사

### **Zero-shot Prompting**

- **한 줄 정의:** 모델에게 별도의 예시를 주지 않고 지시사항만으로 즉시 답을 얻는 기법.
- **핵심 원리:** 대규모 언어 모델(LLM)이 사전 학습(Pre-training) 과정에서 습득한 방대한 지식과 언어 패턴을 활용합니다. Autoregressive 관점에서 보면, 입력된 지시어 다음에 올 가장 확률 높은 '정답' 토큰을 내부 지식만으로 생성해내는 것이다.
- **사용 예시:** "의료급여 1종 수급권자의 입원 본인부담률은?"이라는 질문만 던지는 경우
- **효과:** 간단하고 보편적인 지식 추출에 좋으나, 복잡한 규칙이나 특수한 도메인 지식(우리 과제의 예외 조항 등)에서는 환각(Hallucination)이 발생하기 쉽다.

### **Few-shot Prompting**

- **한 줄 정의:** 프롬프트 내에 해결하려는 문제와 유사한 몇 개의 '질문-답변' 쌍을 예시로 제공하여 모델을 가이드하는 기법.
- **핵심 원리:** 모델의 가중치를 업데이트(Fine-tuning)하지 않고도, 컨텍스트(In-context Learning) 내에서 패턴을 파악하게 합니다. 모델은 예시의 **출력 형식(Format)**과 **추론의 방향성**을 확률적으로 모방하게 됩니다.
- **사용 예시:** 질문: 1종 수급권자인 5세 미만 아동이 상급종합병원에 입원하여 진료를 받으면 본인부담률은 얼마인가요? 답변: "answer": "무료", 같은 예시를 주어 출력 형식이나 사고 과정을 학습시키는 방식.
- **효과:** 특정 도메인의 포맷을 맞추거나 모델이 길을 잃지 않게 할 때 강력하지만, 너무 많은 예시는 컨텍스트 윈도우를 낭비하고 예시 자체에 편향(Bias)될 위험이 있다.

### **Chain-of-Thought (CoT)**

- **한 줄 정의:** 모델이 최종 정답을 내놓기 전에 중간 추론 단계(Intermediate steps)를 거치도록 유도하는 기법.
- **핵심 원리:** 복잡한 문제를 작은 단위로 쪼개어 해결하게 함으로써, Autoregressive 모델이 한 번에 너무 먼 미래의 토큰(정답)을 예측해야 하는 부담을 줄어든다. 중간 단계의 토큰들이 다음 토큰 생성을 위한 강력한 '힌트'가 되어 논리적 오류를 방지한다.
- **사용 예시:** "먼저 종별을 확인하고, 그 다음 연령을 확인하세요"라고 사고의 순서를 지정해 주는 방식.
- **효과:** 논리 연산, 수학, 다단계 조건 판단이 필요한 문제에서 정답률을 드라마틱하게 높인다. 반면, 단순 사실 확인 문제에서는 오히려 비용과 시간만 낭비할 수 있다.

### **Self-Consistency**

- **한 줄 정의:** 동일한 질문에 대해 CoT를 여러 번 수행하여 다양한 추론 경로를 생성하고, 가장 많이 나온 답변을 다수결로 선택하는 기법.
- **핵심 원리:** LLM의 확률적 생성 특성을 활용합니다. 논리적 오류는 경로마다 다를 수 있지만, 정답으로 향하는 논리적 길은 공통적일 확률이 높다는 통계적 원리에 기반한다.
- **사용 예시:** 온도(Temperature)를 높여 5번 답변을 생성하게 한 뒤, 그중 3번 이상 나온 "5%"를 최종 답으로 채택한다.
- **효과:** CoT의 단일 경로 오류를 보완하여 정답의 안정성을 극대화한다. 하지만 비용이 여러 배로 들고 지연 시간(Latency)이 길어진다.

## 2. 추가 기법 조사 (ReAct & Tree of Thoughts)

- **ReAct (Reason + Act):** 모델이 추론(Reason)만 하는 것이 아니라 외부 도구(Search, API 등)를 사용하기 위한 행동(Act)을 계획하고 그 결과를 다시 관찰(Observation)하여 다음 단계로 넘어가는 기법. "생각하고 행동하고 관찰하기"의 반복입니다.
- **Tree of Thoughts (ToT):** CoT가 하나의 길로만 간다면, ToT는 여러 갈래의 생각(Thoughts)을 나무 가지처럼 뻗어보고, 각 생각이 유망한지 평가하여 최적의 경로를 탐색(DFS/BFS)하는 기법. 아주 복잡한 전략적 계획이 필요할 때 쓰인다.

## 3. 실습 과제 예측 및 가설

### **가설 1: Zero-shot(Step 1)의 예상 정답률 및 이유**

- **예측 정답률:** **약 70~80%**
- **이유:** Gemini 2.5 flash 같은 최신 모델은 한국어 이해도가 매우 높고 평상시 사용할 때도 사진을 주어도 데이터를 잘 이해하는 것을 봤던 경험이 있었다. 그렇기에 표 데이터를 텍스트로 주었을 때 기본 문해력이 좋을 것으로 예상했다. 하지만 '만성질환자', '6세 미만' 등의 예외 조항이 겹치는 **다중 조건 필터링**에서 논리적 스텝을 건너뛰어 오답을 낼 확률이 높기 때문에 이렇게 예측했다.

### **가설 2: 가장 큰 개선이 일어날 난이도**

- **예측:** **Hard 난이도**
- **이유:** Easy는 단순 조회라 이미 Zero-shot에서 다 맞을 확률이 높고, Medium은 한두 개의 조건만 추가된다. 반면 **Hard**는 실제 금액 계산(`진료비 * 본인부담률`)과 예외 조항이 섞여 있어, 프롬프트 엔지니어링(특히 CoT)을 통해 계산 과정을 명시할 때 정답률이 비약적으로 상승할 것이라 예측했다.

### **가설 3: 가장 큰 정답률 점프(Jump) 구간**

- **예측:** **Few-shot → CoT (Step 2에서 Step 3로 넘어갈 때)**
- **이유:** 이번 실습의 오답 패턴을 보면 모델이 '규칙'을 몰라서 틀리기보다는 **'규칙을 적용하는 순서'**에서 헷갈려 하는 경우가 많을 것 같았다. Few-shot은 정답의 모양(껍데기)을 잡아주지만, CoT는 정답을 찾아가는 지도(알고리즘)를 주기 때문에 가장 유의미한 점프가 일어날 것으로 예측했다.

class CoTResponse(BaseModel):
    step_1_category: str = Field(description="수급권자 종별 확인 (1종 또는 2종)")
    step_2_age_check: str = Field(description="연령 특례 해당 여부 확인")
    step_3_is_chronic: str = Field(description="만성질환자라는 명시적 조건이 있는지 확인")
    step_4_condition_check: str = Field(description="질환(치매, 조현병 등) 또는 항목(틀니, 추나 등) 특례 확인")
    step_5_institution_check: str = Field(description="의료기관 종별(1, 2, 3차) 확인")
    reasoning: str = Field(description="위 단계들을 종합한 최종 판단 근거")
    answer: str = Field(description="최종 정답 (예: 5%, 무료, 10,000원, 병원급 이상 10% 등)")

system_prompt = f"""
    당신은 의료급여 본인부담률 산정 전문가입니다.
    질문에 답하기 전, 반드시 다음 4단계 과정을 거쳐 사고하세요:

    1단계: 수급권자가 1종인지 2종인지 파악한다.
    2단계: 연령 특례(1세 미만, 6세 미만, 15세 이하 등)가 있는지 확인한다.

    4단계: 특정 질환(조현병, 치매 등)이나 특정 시술(틀니, 임플란트, 추나 등)에 대한 예외 규정이 있는지 확인한다.
    5단계: 방문한 의료기관(1차, 2차, 3차)에 따른 차등이 있는지 확인한다.
    답변이 두개일 시 틀니 -원, 임플란트 -원 같이 간결하게 작성한다.
    적용되지 않을때는 "해당되지 않음" 으로 답변한다.

    최종 답변(answer)은 이전과 같이 불필요한 수식어 없이 '결과값'만 적으세요.

```

            continue

    if not candidate_answers:
        return {"answer": "Error", "candidates": [], "reasoning": "모든 시도 실패"}

    # 다수결 투표
    vote_count = Counter(candidate_answers)
    final_answer = vote_count.most_common(1)[0][0]

    return {
        "answer": final_answer,
        "candidates": candidate_answers,
        "reasoning": f"총 {len(candidate_answers)}회 성공 중 최다 득표 채택"
    }
```

- **방식:** `temperature=0.7` 설정 후 동일 질문에 대해 5회 생성, 다수결(Majority Vote) 투표 적용
- **결과:** 정답률 **100%** (30/30) - 단일 추론의 우연한 실수 보정

### 3-5. 정답률 개선 과정 및 인사이트

- **기법 간 시너지:** Step 2가 '정답의 모양'을 잡아주었다면, Step 3는 '정답을 찾는 논리'를 제공했습니다. 특히 이번 실습처럼 예외 조항이 많은 데이터셋에서는 **사고의 순서(CoT)**를 지정해 주는 것이 모델 체급을 극복하는 가장 효과적인 방법임을 확인했다.
- **구조화된 출력의 중요성:** Pydantic을 활용해 사고 단계를 필드로 정의하니, 모델이 해당 필드를 채우기 위해 강제로 관련 정보를 한 번 더 훑게 되어 실수(Skipping)가 현저히 줄어들었다.
- **답변 형식에 대한 고찰** 예측에서는 인공지능이 우선순위 판단을 어려워 할 것으로 예상하였지만, 오히려 정답 그 자체를 맞추는 것에는 크게 어려움이 없었다. 하지만 답변이 원하는 정확한 문구로 답을 도출하는데 어려움이 있었다. 실제 업무에서도 원하는 답변 구조안에서 답변을 받아야 할텐데 어떻게 하면 인공지능이 다양한 답변 형식 중에 내가 원하는 형식으로만 답변할 수 있을지 고민해 보아야 할듯 하다.
